SECTION D: INCIDENT ONCE-ONLY FLAG SYSTEM
============================================================

Every incident in incident-bank.md carries two tracking fields:

  "Told in" = the ONE chapter where the incident is told in full.
              Only one chapter may own the full telling. Ever.

  "Refs"    = brief mentions in other chapters (1-2 sentence
              callback, echo, or image only — never re-telling).
              MAXIMUM 2 refs per incident across the entire book.

AGENT RULES:
  - Before placing an incident in full, check "Told in" is empty
  - Before adding a brief reference, check "Refs" count < 2
  - An incident with "Told in" filled AND 2 refs is CLOSED
  - When placing or referencing, update the bank IMMEDIATELY
  - During reorganization, "Told in" may change — but the
    once-only rule still applies (just to a different chapter)

SECTION E: SCRATCHPAD INLINE MARKERS (@@ SYSTEM)
============================================================

Every new scratchpad file MUST have TWO layers of marker visibility:

LAYER 1 — FULL REFERENCE BLOCK (top of file, before Section 1):

  === @@ QUICK MARKERS — drop at start of any line ===
  @@refine  — sharpen this sentence or paragraph
  @@replace — find a stronger word
  @@expand  — flesh out, add detail, slow down
  @@cut     — remove this, it does not earn its place
  @@move    — relocate (add hint: @@move to section 3)
  @@note    — message to Claude, not content
  === END MARKERS ===

LAYER 2 — COMPACT REMINDER (pinned under every section header):

  === SECTION N: Title ===
  @@ refine | replace | expand | cut | move | note

This one-liner appears directly below EVERY section header so Asif
always sees the available markers no matter where he is scrolling.
Both layers are stripped during finalization.

AGENT RULES FOR @@ MARKERS:
  - On every rebuild or review pass, scan the scratchpad for @@ markers
  - Process each marker according to its type
  - After processing, REMOVE the @@ marker from the line
  - If Asif left a @@note, read it, act on it, strip it
  - Never leave processed markers in a finalized chapter
  - The marker block header is stripped during finalization

SECTION F: INCIDENT INTAKE PROTOCOL
============================================================

TRIGGER: Any of the following:
  - Asif says "intake" or drops a list of rough incidents
  - Asif drops a single incident (one phrase, one sentence)
  - Asif drops a CATEGORIZED or UNCATEGORIZED list
  - Short phrases, sentence fragments, dashes — any format

RECOGNIZED INPUT FORMATS:

  FORMAT A — SINGLE INCIDENT (no list, no category)
    One incident dropped as a phrase or sentence:
      "the time Baba slapped me at the dinner table"
      "Amma reading my diary"

    RULES:
      - Treat as a batch of one
      - Run qualifying questions for this single incident
      - Log to incident-bank.md like any other

  FORMAT B — FLAT LIST (multiple incidents, no category)
    Multiple incident skeletons with no grouping header:
      - something happened at school
      - the time Baba did X
      - Amma and the phone call

    RULES:
      - Agent infers likely chapter/era from biographical context
      - Groups them by likely chapter in the qualifying round
      - If chapter assignment is unclear, ask Asif

  FORMAT C — SINGLE CATEGORY WITH LIST
    One category header followed by its incident skeletons:

      Marriage with ex wife
      * she demanded X while drunk
      * the night I slept in the car
      * her mother showed up unannounced

    RULES:
      - The header is NOT an incident — it is a grouping label
      - Use the header to auto-fill: likely chapter tag, era,
        and relationship context for every incident beneath it
      - If the header maps clearly to a chapter (e.g., "Marriage
        with ex wife" → Ch03-Marriage), pre-tag all incidents
      - If ambiguous, ask Asif which chapter before logging

  FORMAT D — MULTIPLE CATEGORIES WITH LISTS
    Multiple category headers, each with their own incidents,
    dropped in a single message:

      Marriage with ex wife
      * she demanded X while drunk
      * the night I slept in the car

      Childhood in Kuwait
      * the playground fight
      * Baba's Friday routine

      Ishrat
      * the first Ramadan together
      * when she called my bluff

    RULES:
      - Each header is its own context bucket
      - Process ALL categories in a single batch
      - Each header auto-fills chapter tag, era, and context
        for its child incidents independently
      - Qualifying questions are grouped BY CATEGORY so Asif
        answers in clusters, not one-by-one
      - Categories may map to the same chapter (e.g., "Marriage
        with ex wife" and "The cousin" both → Ch03-Marriage)

  CATEGORY HEADER RULES (apply to Format C and D):
    - The header is NEVER logged as an incident
    - Use the header to auto-fill: likely chapter tag, era,
      and relationship context for every incident beneath it
    - If the header maps clearly to a chapter, pre-tag all incidents
    - If ambiguous, ask Asif which chapter before logging
    - Headers can be: a person (Amma, Ishrat, ex-wife), an era
      (childhood, college, first marriage), a theme (faith, money),
      or a chapter name (Marriage, Man, Love)

AGENT WORKFLOW:

  STEP 0 — PARSE FORMAT
    - Detect which format: A (single), B (flat list), C (one category),
      or D (multiple categories)
    - For A: treat as batch of one, proceed to Step 1
    - For B: infer likely chapter/era from biographical context
    - For C/D: extract each category header and its child incidents,
      map headers to chapter tags and era using biographical context
      and thematic arc
    - Carry all context forward into qualifying questions so Asif
      does not repeat what the header or context already provides

  STEP 1 — BATCH SCAN
    - Read the full incident-bank.md
    - Compare Asif's list against existing incidents
    - Flag overlaps ("this sounds like INC-025, confirm?")
    - For Format B: group new incidents under their category header
    - For Format A: group new incidents by likely chapter

  STEP 2 — SMART QUALIFYING ROUND (one pass, not one-by-one)
    - For the ENTIRE batch, ask ONE consolidated set of questions
    - ALWAYS use AskUserQuestion tool with selectable options
    - Group questions by category so Asif answers in clusters
    - The category header already answers "which chapter" and
      "what era" — do NOT re-ask what the header already provides
    - Standard question types (all as selectable lists):
        "Where does this land emotionally?"
          → (a) anger (b) sadness (c) humor (d) quiet realization
        "How old were you?"
          → (a) childhood (b) teens (c) twenties (d) later
        "Should this connect to an existing scene?"
          → (a) yes — which one? (b) no — standalone
    - If an incident is clear enough from context, skip questions for it
    - NEVER guess or invent details. Ask if unsure.

  STEP 3 — DRAFT TO INCIDENT SCRATCHPAD
    - Create or open: JOURNAL_DIR/scratchpads/scratchpad-incidents.txt
    - This file contains ONLY refined incident prose, nothing else
    - Each incident is a NUMBERED SECTION with @@ markers:

        === INCIDENT 1: [Short Title] ===
        @@ refine | replace | expand | cut | move | note

        [Refined prose in Asif's voice, based on his skeleton
         and qualifying answers. Voice DNA fingerprint applied.
         No invented details. No em dashes. No therapy jargon.]

        === INCIDENT 2: [Short Title] ===
        @@ refine | replace | expand | cut | move | note

        [Next incident prose...]

    - Run all five quality loops on each incident draft
    - ALWAYS present the file to Asif using a computer:// link:
        [Review your incidents](computer:///sessions/clever-zealous-hamilton/mnt/Journal/scratchpads/scratchpad-incidents.txt)
    - This is NON-NEGOTIABLE. Every time the scratchpad is created
      or updated, open the file for Asif. He reviews in the file,
      not in chat. The link is how he gets there.

  STEP 4 — INTERACTIVE REVIEW (same pattern as chapter workflow)
    - Asif reads one incident at a time
    - He can say or mark:
        "Incident 2, change xyz" → update in place
        "Incident 3 is good" → mark approved
        @@refine on a line → sharpen that sentence
        @@expand → add detail, slow down
        @@cut → remove, does not earn its place
        @@replace → find a stronger word
        @@move → relocate within the incident
        @@note → message to Claude, not content
    - After processing @@ markers, REMOVE them from the line
    - Iterate until Asif approves each incident

  STEP 5 — APPROVE AND LOG TO INCIDENT BANK
    - ONLY when Asif explicitly approves an incident:
      1. Assign next sequential INC number
      2. Log to incident-bank.md with:
           Title, Chapter tag, Era/age, Themes, Emotional arc position,
           Status (BANKED), Connections, Takeaway
      3. The refined prose from the scratchpad becomes the incident's
           canonical text (referenced when placing into a chapter later)
    - Confirm to Asif: INC number assigned, chapter target, status
    - Remove approved incidents from scratchpads/scratchpad-incidents.txt
    - When scratchpad is empty (all approved), delete the file

  STEP 6 — ROUTE
    - If an incident clearly belongs in a LOCKED chapter, log it but
      mark: "BANKED — chapter locked, will place during reorganization"
    - If it belongs in the active chapter, offer to draft it into the
      chapter scratchpad immediately
