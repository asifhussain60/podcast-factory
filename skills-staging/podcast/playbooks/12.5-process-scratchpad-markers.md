# Stage 12.5 — Process Scratchpad Markers

**REQUIRED READING:** `journal/reference/scratchpad-markers.md` — the canonical specification of the `@@` marker vocabulary, shared with the `journal` skill. Read it once per session before processing any scratchpad.

**Purpose:** Apply the user's directives from scratchpad files to their corresponding refined episodes, then strip the processed markers so the scratchpad does not accumulate stale directives. Runs after Stage 12 (refinement) on any pass where a scratchpad exists.

**When this stage runs:** Anytime a scratchpad file exists for a refined episode and contains user-added `@@` markers. Skipped silently when no scratchpad exists or when the scratchpad contains only its legend block and per-section one-liners.

## Input

- `WORK_DIR/scratchpad/episode-NN-<slug>.scratch.md` — the user's marked-up parallel file
- `WORK_DIR/01-refined/episode-NN-<slug>.md` — the canonical refined episode
- `journal/reference/scratchpad-markers.md` — vocabulary spec
- `WORK_DIR/03-pronunciation.md` — referenced by `@@pronounce` markers

## Output

- Updated `WORK_DIR/01-refined/episode-NN-<slug>.md` reflecting all marker directives
- Updated `WORK_DIR/scratchpad/episode-NN-<slug>.scratch.md` with processed markers stripped (legend and per-section one-liners preserved)
- `WORK_DIR/_meta/marker-manifest.md` — log of every marker processed in this pass: file, line, verb, args, action taken

## The 11 verbs

Documented canonically in `journal/reference/scratchpad-markers.md`. For quick reference, grouped by propagation tier:

### Tier 1 — Local (apply only at marker location)

| Verb | Effect |
|---|---|
| `@@refine` | Sharpen this sentence or paragraph in place. |
| `@@replace` | Find a stronger word for the marked term. |
| `@@expand` | Flesh out, slow down, add detail. |
| `@@cut` | Remove the marked content. |
| `@@move(destination)` | Relocate the marked content to the destination. |
| `@@note(message)` | Read the message, act on it if actionable, strip. |
| `@@merge(direction)` | Combine the marked paragraph or section with its neighbor. |
| `@@rephrase` | Propose 2-3 alternate phrasings; the first becomes the canonical replacement. |
| `@@split(hint)` | Break the marked paragraph or section into two. |

### Tier 2 — Mechanical (auto-propagate with one-line confirmation)

| Verb | Effect |
|---|---|
| `@@pronounce(term: phonetic)` | Override pronunciation for the term throughout the series. On detection, prompt user to propagate to every other episode in the project. The series-wide pronunciation guide (`03-pronunciation.md`) is NOT touched — this is an override scoped to the run's refinement passes. |

### Tier 3 — Policy (apply as Hard Rule augmentation to every future refinement)

| Verb | Effect |
|---|---|
| `@@policy(directive)` | Series-wide style/voice/structural/lexical rule. Lifted into `scratchpad/series-policies.md`, augments Stage 12 Hard Rules on every refinement pass thereafter. Examples: `@@policy(reduce formality across episodes — more conversational, fewer ceremonial connectors)`, `@@policy(every Quranic transliteration's English meaning lands on its own line)`. |

## Processing flow

### Step 1 — Scan and classify

Read the scratchpad. Identify every line containing `@@` markers that are user-added (not part of the legend block or the per-section one-liner). For each:
- Record file, line number, verb, parsed argument.
- Identify the **tier** by verb (Tier 1 local / Tier 2 mechanical / Tier 3 policy).
- For Tier 1, identify the **scope** — does the marker apply to the line itself, the paragraph below it, the whole section, or the entire episode?

Scope inference (Tier 1 only — Tier 2 and Tier 3 are inherently series-wide):
- Marker on its own line above a paragraph → scope is the paragraph below.
- Marker at the start of a line followed by content → scope is the line itself.
- Marker on a section header line → scope is the whole section.
- Marker as the first line of the file (above the canonical content) → scope is the entire episode.

### Step 1.5 — Detect propagation candidates

After classification, present the user with a one-line summary if any Tier 2 or Tier 3 markers were found:

```
Detected in episode-01 scratchpad:
  • 7 local edits (will apply to this episode only)
  • 2 pronunciation overrides (@@pronounce) — propagate to remaining 6 episodes? [Y/n]
  • 3 series policies (@@policy) — record in series-policies.md and apply to remaining 6 episodes? [Y/n]
```

If user accepts:
- Pronunciation overrides get applied to every episode in the run during their refinement pass.
- Policies get recorded in `WORK_DIR/scratchpad/series-policies.md` with provenance (which scratchpad, which date) and become Hard Rule augmentation for Stage 12 on every subsequent episode.

If user declines (Y/n → n), the Tier 2 / Tier 3 markers are still applied to *this* episode but not propagated. They are then stripped from the scratchpad as usual — they won't trigger the prompt again on subsequent runs.

### Step 2 — Validate

For each marker:
- Verb must be in the 10-verb set per `scratchpad-markers.md`. If unknown, halt and write `MARKERS-UNKNOWN-VERB.md` listing the offending lines.
- Argument format must match the verb's expected pattern (`@@pronounce` requires `term: phonetic`; `@@move` requires a destination; etc.).
- Scope must be resolvable. If a marker has ambiguous scope, halt and ask the user.

### Step 3 — Process

Apply each marker in document order. For content-mutating verbs:
- **refine** — rewrite the scoped content per Stage 12 Hard Rules (propositional content preserved, articulation free). Single rewrite.
- **replace** — for the scoped word or phrase, propose a stronger replacement. Drop the replacement in place.
- **expand** — increase the scoped content's word count by ~30-60%, adding detail, dialogue, or analogy where the source supports it. Must not exceed Gate 5 word-count ratios.
- **cut** — delete the scoped content. Adjust surrounding sentences for continuity.
- **move(destination)** — remove from current location, insert at destination. Update any cross-references.
- **note(message)** — read the message. If it asks for a specific change, apply it; if it asks a question, answer in the marker manifest and flag for user review; if it's a reminder, log and proceed.
- **merge(direction)** — combine paragraphs or sections as directed.
- **rephrase** — generate 2-3 alternate phrasings; the first becomes the canonical replacement; the other 1-2 go into the marker manifest for user reference.
- **split(hint)** — break the scoped paragraph or section into two pieces at the indicated point.
- **pronounce(term: phonetic)** — record the override. If user accepted propagation in Step 1.5, the override applies during every other episode's next refinement pass. Do NOT change `03-pronunciation.md` (the series-wide reference) unless the user has explicitly asked.
- **policy(directive)** — append to `WORK_DIR/scratchpad/series-policies.md` with provenance (this scratchpad, this date). If user accepted propagation in Step 1.5, the directive becomes Hard Rule augmentation for Stage 12 on every subsequent episode refinement. Idempotent: re-marking an existing active policy is a no-op.

### Step 4 — Strip

For every marker successfully processed, remove the marker from the scratchpad. The line containing the marker stays (its content was preserved or rewritten); only the `@@verb(...)` token and any trailing whitespace are removed. The legend block at the top of the scratchpad is preserved. Per-section one-liners (`@@ refine | replace | expand | ...`) are preserved.

### Step 5 — Validate strip

Re-scan the scratchpad. Confirm zero remaining user-added `@@` markers. The only `@@` content should be the legend block and the per-section one-liners. If anything else remains, that's a defect — halt and report.

### Step 6 — Write the manifest

Write `_meta/marker-manifest.md` for this pass:

```markdown
# Marker manifest — pass 2026-MM-DD HH:MM

Episode: episode-01-frame-and-first-counsel

| # | Line | Verb | Tier | Args | Scope | Action |
|---|---|---|---|---|---|---|
| 1 | 47 | refine | local | — | paragraph | Rewrote: "He carries skilled bows" → "He has skilled bows in hand" |
| 2 | 62 | pronounce | mechanical | Sunnah: Soon-nah | line | Override applied for this episode + propagated to 6 remaining episodes |
| 3 | 78 | note | local | check verse 18:110 reference | — | Verified — verse number correct per Hasan translation |
| 4 | 91 | expand | local | — | paragraph | Expanded "He never takes it" passage by 40 words; ratio now 0.85 |
| 5 | 105 | cut | local | — | sentence | Removed redundant restatement of the lion analogy |
| 6 | 8 | policy | policy | reduce formality across episodes | series | Recorded in series-policies.md; applies to remaining 6 episodes |

All 6 markers processed and stripped. Local: 4. Mechanical propagated: 1. Policies recorded: 1. Validator pass: OK.
```

## Hard rules

1. **Never apply a marker change that would breach Stage 14 quality gates.** If a marker directive would push the episode outside the 1,800-2,800 target band or outside the 70-140% source-ratio, halt and ask the user.
2. **Never modify the legend block or the per-section one-liners.** These are persistent reference for the user.
3. **Never touch `03-pronunciation.md`** in response to `@@pronounce`. The marker is an override scoped to the run's refinement passes. To change the series-wide pronunciation guide, the user edits it directly or files a library proposal.
4. **Never re-segment the episode in response to a single marker.** Segmentation is governed by `segments.yml`. Markers can request `@@cut` or `@@expand` or `@@merge` within an episode, but they cannot move content across episodes — that requires a `segments.yml` edit.
5. **One marker pass per refinement run.** Do not loop on marker processing. If the user wants iterative refinement, they re-mark the scratchpad after seeing the result.
6. **Tier 2 / Tier 3 propagation requires explicit user confirmation.** Never propagate `@@pronounce` or `@@policy` across episodes without the one-line Y/n prompt described in Step 1.5. The user always has the chance to keep a directive scoped to one episode.
7. **Policies are additive, never destructive.** A `@@policy` directive augments Stage 12 Hard Rules; it cannot remove or weaken them. A policy that says "more conversational" still operates inside Hard Rule 5 (propositional content preserved). The hard rules are the hard floor — policies live above them.
8. **Policies are idempotent.** Re-marking an already-active policy in a new scratchpad does not duplicate the entry. The skill checks `series-policies.md` and silently skips duplicates with a one-line log.

## What this stage does NOT do

- Does not generate the per-episode instruction blocks (Stage 13).
- Does not re-run quality gates (Stage 14 — runs after this stage on a normal pipeline).
- Does not produce the ZIP (Stage 15).
- Does not modify the master lexicon, library files, or `segments.yml`.
