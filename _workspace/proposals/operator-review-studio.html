Section 4

Six features that compress your reading time

These are the design moves that turn 45 min into ~8 min. Each can be turned off if it's noise.

Anchored flagging

Select any text in the transcript → click Flag this. A row appears in the review form with the page number and the quoted text pre-filled. You just type the fix. No more "wait, what page was that?"

Fuzzy search + page jump

⌘F finds any phrase across the transcript. ⌘P jumps to a page by number. Together they replace scrolling for ~80% of operator workflows — "where did the second-author voice shift happen again?"

Diff-with-raw toggle

Switch on Diff with raw and the transcript pane highlights every word the LLM changed during Phase 0b refinement. Now you only re-read the parts the model touched — the rest is identical to the OCR output. Massive reading-time win on long books.

Sticky page indicator + outline

A small floating badge always tells you what page you're on (synced as you scroll). A right-side outline shows §-headers, flags placed, and how far through you are. You always know where you are and what's left.

Auto-save + crash-safe drafts

Every keystroke saves a draft to your browser's storage. The actual operator-review.md file updates every few seconds (debounced). Close the tab, reboot your Mac — your work is still there when you come back.

One-click Approve & Resume

The big purple button at the bottom does what you do today by hand: writes your approval, then runs the resume command for that book. You watch the orchestrator log scroll in a side drawer. No Terminal switch.

Phase B — if you want them later
AI per-page summaries. Optional Haiku call (~$0.05/book) emits a 1-line summary per page — skim mode.
Sentence-level OCR confidence. Lighter color = lower confidence. Read low-confidence sentences first.
Resume orchestrator log drawer. See live Phase 0c output stream after you hit Approve, without Terminal.

Section 7

Decisions I need from you

Six questions. Answer in whatever order. I won't write a line of app code until these are settled.

Q1

React or Angular?

My recommendation: React 18 + TypeScript + Vite. React has the larger ecosystem for the components I need (virtualization, markdown rendering, fuzzy search). Angular works fine but adds scaffolding you don't need for a focused tool.

Default if you say nothing: React.

Q2

Tiny FastAPI backend, or "no backend, browser only"?

A browser SPA can't directly write to your local operator-review.md without either (a) a tiny local server, or (b) the File System Access API (Chrome-only, asks permission each session). My recommendation: FastAPI backend on localhost:8766, same pattern as the P8 dashboard.

Default: tiny FastAPI backend.

Q3

One worktree at a time, or all worktrees in the same view?

You have three worktrees right now (Journal, Journal-book-asaas, Journal-feat-w1). The MVP could either: (a) one server per worktree, or (b) one server that lists books across all known worktrees. Option (b) is the long-term answer but more complex.

Default: (a) one worktree at a time. Promote to (b) later.

Q4

Should the app actually fire --approve-transcript for you?

This is the "auto-resume" feature on the Approve button. Some operators prefer to run the resume command themselves so they can pick the right worktree, watch the log, etc. The app can either: (a) fire it directly and show the log; (b) copy the command to your clipboard and let you run it.

Default: (a) fire it AND show the log in a drawer. Cancellable.

Q5

Phase A only, or A + B + C in one build?

Phase A (MVP, ~1 day): books list + dual-pane editor + auto-save + approve.
Phase B (~1 day): anchored flagging, fuzzy search, page jump, sticky page indicator.
Phase C (~1 day): reading checklist, diff-with-raw, resume log drawer.
Phase D (later): AI summaries, OCR confidence shading, dashboard merge.

Default: ship A + B together (the two-pane editor without anchored flagging is too lossy). Hold C until A+B is used in anger.

Q6

Track as new plan phase P25 in the YAML?

This is a real new feature with its own deliverables, tests, and acceptance criteria. Tracking it as P25 — Operator Review Studio in _workspace/plan/podcast-plan.yaml (parallel to P22, P23, P24) makes it visible to the autonomous loop and to anyone reviewing the plan.

Default: yes, add P25 to the plan before writing code.
