The core idea, in one sentence

After phase 0g audit completes for a book, an Extractor reads its audit-vetted chapters and pulls out atoms (verses, hadith, quotes, etc.); a Librarian compares them to a canonical library and dedupes; an Augmenter reads from that library during future books' phases so the model writes richer chapters by knowing what's been said before. The first book pays in; later books cash out.

What gets captured

Five atom types radiate from one canonical library. Each type has a different shape and a different dedup strategy — that's the whole reason this is interesting.

Five atom types, one library

Mind map

Two atom types (Quran, hadith) have fixed canonical IDs — exact-match dedup is a two-line function. Two (quotes, definitions) need embedding similarity because the same idea appears in many wordings. Etymology is its own shape — a root-keyed tree.

Three pieces, not one big agent

Each piece runs on its own and is testable in isolation. No monolithic "intelligence agent." Each has one job, one input, one output.

Piece 1

Extractor

Runs end of phase 0g audit, per book

Reads every audit-vetted chapter. Pulls out atoms. Doesn't ask if it's seen them before — it just collects.

Librarian

Runs right after Extractor

Takes the scratch file. Walks each atom: new? known? near-duplicate? Writes back to the canonical library or flags conflicts.

In _system/knowledge-atoms-scratch.jsonl

Piece 3

Augmenter

Runs during 0e enrich + per-chapter + 0g audit of future books

Before the model writes about a topic, queries the library for relevant atoms and slips them into the prompt. The model writes richer with context. Current book gets no benefit — only books that follow.

Extractor → Librarian → Augmenter data flow

Data flow

The book's own branch writes to the library, just like everything else under content/. The Augmenter reads at query time — no special sync. Per book, the Librarian step ends with a merge report ("X new, Y updated, Z flagged"); flagged entries halt phase 0h until reviewed.

Where it slots into the orchestrator

A new phase 0h sits between 0g (audit) and finalize. It owns both Extractor and Librarian. The Augmenter is not a phase — it's a query helper invoked from inside other phases of future books.

Why 0h, not 0g

Phase 0g is already taken — it runs the dual-auditor (audit_bundle.py + audit_bundle_gemini.py) per-chapter audit after authoring. The new knowledge-extract phase slots in as 0h, right after audit but before finalize, so atoms come from audit-vetted content.

Phase order with 0h inserted

Flowchart
future books' phases: · 0e enrich — inject prior treatment context early · per-chapter + 0g audit — full cross-book context

0h halts on conflicts before finalize, so we catch bad atoms before a book ships. The Augmenter is purposely not a phase — it's a stateless lookup that any phase of a future book can call. Current book's content is already locked by the time 0h runs, so the Augmenter benefits only books that follow.

What an atom looks like

Every atom shares a common envelope (id, type, sources, variants, embedding). The body differs per type. JSONL — one line per atom, greppable, diff-friendly, git-mergeable.

① Quran verse

id: quran:<surah>:<ayah>

Body: surah, ayah, text_ar, text_en, optional tafsir_refs

Easy · id match

② Hadith

id: hadith:<collection>:<num>

Body: collection, number, grade, text_ar, text_en, chain

Easy · id match

③ Quote

id: quote:<sha256(normalized)>

Body: attributed_to, text_ar, text_en, context

Fuzzy · embedding sim ≥ 0.92

④ Etymology

id: etym:<root>

Body: root, derivations[], meaning_chain; children attach as nested entries

Tree · root + child match

⑤ Definition

id: def:<term>:<domain>

Body: term, domain (fiqh, kalam, tasawwuf...), definition, references[]

Fuzzy · embedding sim ≥ 0.92

Dedup strategy per atom type

The whole point of the Librarian: don't store the same thing twice. Different types need different dedup logic. Two atom types are trivial; three are not.

Dedup difficulty by type

Control matrix

Embedding model: text-embedding-3-small (cheap, good Arabic+English coverage, $0.02 per 1M tokens). Embeddings stored alongside atoms; index rebuilt incrementally by the Librarian. Atoms in the 0.80–0.92 similarity band go to _conflicts/pending-review.jsonl for human adjudication before merge.

Where the library lives

One folder, plain text files, version-controlled. No database to manage, no daemon to keep running. Greppable from the command line, readable in a diff viewer.

content/ └── knowledge-base/ ├── quran.jsonl # one atom per line ├── hadith.jsonl ├── quotes.jsonl ├── etymology.jsonl ├── definitions.jsonl ├── _index/ │ ├── embeddings.faiss # vector index, rebuilt by Librarian │ └── stats.json # counts, last-updated, top-cited └── _conflicts/ └── pending-review.jsonl # flagged near-duplicates awaiting your call

Why JSONL, not SQLite or a vector DB

JSONL gives git-friendly line-level diffs, is grep-friendly from any shell, requires no runtime daemon, and survives a five-year shelf life. SQLite buys query speed we don't need until we have hundreds of thousands of atoms. The FAISS index is the only binary artifact and is rebuildable from the JSONL alone — if it gets corrupted, delete and regenerate. No "database backup" discipline required.

The flywheel

Each book through the pipeline makes the library richer; each richer library makes the next book's enrichment + authoring smarter. The first book gets no benefit. The hundredth book walks in on the shoulders of ninety-nine prior treatments.

Compounding intelligence loop

Lifecycle

The flywheel asymmetry is real: there's a cold-start period (books 1–3 get little augmentation value). The pay-off accelerates as the library densifies. Worth pricing in when judging early-book "did this help?" — answer is "ask me again at book 5."

Risks named upfront

The things that could go wrong, and what we'll do about them before they do.

Embedding cost creep

Every new atom gets embedded. Every Augmenter query gets embedded. Over many books this adds up.

Mitigation: cache aggressively, embed only on add (not on every Librarian run), batch where possible. Cap: $2.00 per book (raised from initial $0.50 after self-review; tafsir-heavy books can hit 200+ citations).

Bad atoms poisoning the library

A garbled hadith reference extracted once can live forever and surface in every future book's prompts.

Mitigation: Librarian flags low-confidence atoms in _conflicts/. Human reviews before they merge to develop with the book.

Augmenter prompt bloat

Injecting 30 prior atoms into every authoring prompt blows the context window and confuses the model.

Mitigation: cap injection at top-K most-relevant (K=5), explicit "use only if directly relevant" framing in the prompt.

Branch-merge conflicts on the library

Two books in flight on parallel branches both try to add quran:2:255 first.

Mitigation: Librarian writes happen at the publish merge step, not mid-book. Conflicts resolve by Librarian re-run at merge time, not by git's auto-merger.

Scope creep into "research tool"

Once the library exists, the temptation to build a web viewer, semantic search UI, RAG chat will be strong.

Mitigation: resist until augmentation is proven valuable on at least 5 books. The library has its own intrinsic value but the UI is a separate, future decision.

Cold-start illusion

Books 1–3 get little augmentation benefit because the library is thin. Tempting to declare the whole thing a failure too early.

Mitigation: set explicit "judge at book 5" checkpoint. Track atom counts per type as the durable success metric, not subjective "did this help?".

Alternatives considered, not chosen

Briefly, for the record. None of these are wrong; they're just not what we picked.

Run extraction post-publish

Would have been simpler — book is fully done, no in-flight state to worry about. Rejected because the per-chapter authoring loop wouldn't be able to read from the library for the current book, losing 50% of the flywheel value.

SQLite + vector DB instead of JSONL

Faster queries at scale. Rejected at this stage because JSONL is git-mergeable, grep-friendly, and survives long-term storage. Revisit at ~100k atoms or if Augmenter query latency becomes a bottleneck.

Build all five types in parallel

Your decision was "all five at once" for scope, and that holds. Build order is still phased (easy → fuzzy → tree) because debugging dedup logic for three squishy types in week 1 is a stall risk. Same end state, lower risk.

Separate repo for the library

Would isolate the library cleanly. Rejected because the Augmenter needs to read from the same workspace that's running the pipeline. Cross-repo sync is more failure modes than benefits at this scale.
