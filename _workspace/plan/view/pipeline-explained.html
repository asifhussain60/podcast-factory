Pipeline Explainer

How the pipeline turns a 260-page Arabic book into 14 podcast episodes

The pipeline takes a scanned Arabic scholarly book in PDF form and produces a series of NotebookLM-ready audio overview episodes. It does this in nine phases — first preparing the text (OCR, translation, polishing, pronunciation), then designing the episode structure, then authoring each episode through an iterative "challenger and rewrite" quality loop until each one is ready to upload.

Running example: this page uses Kitab al-Riyad as a concrete illustration. The generic descriptions on the left of each row apply to every book the pipeline processes — Raahat al-Aqal, Kitab Maqbas, Rasail Ikhwan AsSafa, and others queued behind Kitab al-Riyad.

0a Phase

Ingest the PDF

Done

0b Phase

Refine the English

about $3-5 · ~95 min
Done
Generic — every book

The raw OCR-and-translate output is awkward English with mistranslations, garbled term boundaries, and broken sentence structure. Claude reads it in roughly 3,000-word chunks and rewrites each chunk into polished, faithful English — preserving every page marker so later phases can cite back to the source page. The chunked approach prevents Claude from losing track over long contexts.

0c Phase

Build the Arabic pronunciation guide

about $5-6 · ~3 hours
Done
Generic — every book

Scholarly Islamic books are dense with Arabic terms — names, theological vocabulary, place names. NotebookLM has to read these aloud, and if the pronunciation is wrong the episode sounds amateur. This phase scans the refined English for every Arabic term and authors a phonetic respelling (tawhid → taw-HEED, al-mukhalif → al-moo-KHAH-lif). The phonetic file is consulted later by every episode's framing.

0d Phase

Chapter design

about $4-6 · ~3 hours
Done
Generic — every book

Decide how the book becomes episodes. Arabic scholarly books are organized into abwāb (chapters) and fusūl (sub-chapters). Some abwāb are too long for a single 30-45 minute episode and need splitting; some need bundling. This phase reads the structure and writes a "contract" for each future episode — a YAML file declaring audience, angle, host dynamic, central tensions, and 3-4 verbatim anchor passages the hosts will discuss. The contract is the blueprint for per-episode authoring later (Phase 0g).

What goes inRefined English manuscript + structural map
What comes outOne chapter-contract YAML per planned episode

Source has 10 abwāb. Six were too long for single episodes and got split (resulting in letter-suffix naming — ch01a, ch03a, ch04b, ch05c, ch13a, ch14b indicate sub-episodes of a parent bāb). One bāb was dropped from the episode list (manuscript-history bookkeeping with low listener value). Result: 14 episode contracts.

How an episode actually gets written

Inside Phase 0g, every episode goes through four sub-steps. The fourth — Converge — is the iterative loop that does the heavy quality work.

Step 1
Extract

Pulls the chapter source from the file system into the orchestrator's working memory. Ensures the chapter is enriched (Phase 0e output) before proceeding.

Step 2
Frame

Calls Claude to author the NotebookLM customize prompt. The prompt has ~12 sections (Opening, Background, Audience, Angle, Central tensions, Host dynamic, Three-part focus, Tone, Pronunciation, Anti-noise, Do not, Upload checklist) — each tailored to the chapter's content and the series plan's persona/tier.

Step 3
Build

Runs the validator (build_episode_txt.py) against the chapter source + framing. Checks ~15 rules. If any "hard" rule fails, the episode halts here for a fix before moving on.

Step 4
Converge

Runs the challenger: an independent Claude pass that critiques the episode, surfaces P0/P1/P2 issues, and rewrites. Loops up to 3 outer × 5 inner times. Ships at SHIP-READY (P0=0, P1=0, P2≤2) or SHIP-WITH-CAUTION (P0=0, P1≤3).

Kitab al-Riyad concrete: EP10's convergence path was iter=3 fix=4 P0=0 P1=2 P2=2. It took 3 outer rounds; 4 fixes were applied across them; final state had 2 P1 issues judged acceptable; shipped SHIP-WITH-CAUTION. Each episode is allowed up to 15 convergence passes before halting for human review.

Are the podcaster roles consistent?

Short answer: within a book, yes. Across books, deliberately variable based on audience. A NotebookLM episode has two hosts. The pipeline names them Driver and Color:

Driver

Moves the conversation forward, asks the framing questions, drives toward the central tensions named in the chapter contract.

Color

Provides historical and contextual richness, brings in tangents, offers analogies that make the substance land for the listener.

The persona defines how Driver and Color act

Four common personas. Persona is chosen per book; Kitab al-Riyad uses Mentor+Student.

Mentor + Student

Driver is the experienced scholar, Color is the curious learner. Default for content-dense scholarly works.

Kitab al-Riyad uses this
Peer + Peer

Both hosts at similar expertise level — conversational and exploratory.

Critic + Defender

Driver challenges, Color defends. Used for polemic texts where the audience benefits from seeing the argument tested.

Storyteller + Analyst

Driver narrates events, Color analyzes meaning. Used for biographical or historical-event-heavy books.

The tier defines length

Default (12-20 min) For shorter or less-dense chapters.
Extended (30-45 min) For chapters with substantial theological or philosophical content. Kitab al-Riyad uses Extended.

Consistency rule: persona + tier are declared in the series plan (Phase 0f) and apply to every episode in the book. Changing them mid-book would create a jarring listener experience. Across books, persona is chosen based on the book's specific audience + content type.

How does the pipeline get Arabic pronunciation right?

Three coordinated mechanisms work together. The phonetic guide is built once per book; the framing wires it into the imperative form NotebookLM actually obeys; and a validator rule keeps pronunciation guides out of the chapter source itself.

Step 1

Phase 0c builds the phonetic guide

Every Arabic term gets a phonetic respelling using [STRESSED]-[unstressed] pattern. E.g. tawhid → taw-HEED, al-mukhalif → al-moo-KHAH-lif, nāṭiq → NAA-tiq. Targets natural English-language voice synthesis.

→
Step 2

Framing uses imperative form

Phase 0g writes a ## Pronunciation section with imperative lines: Pronounce "tawhid" as "taw-HEED". Say it as one fluent word. The R-PRONUNCIATION-IMPERATIVE rule enforces this — a passive list ("tawhid: taw-heed") would be ignored by NotebookLM.

→
Step 3

R-PHONETICS-OUT keeps guides out of source

Pronunciation guides MUST live in the framing, not the chapter source. If they leak into the chapter source, NotebookLM reads them aloud (e.g. "Sunnah open-paren SOON-nah close-paren"). The validator rejects any inline guides.

Worked example — Kitab al-Riyad

A chapter on "tawhid" might include Pronounce "tawhid" as "taw-HEED". Say it as one fluent word. in the framing's Pronunciation block. The chapter source itself just uses the word "tawhid" in prose — no respelling. NotebookLM then naturally pronounces "tawhid" correctly whenever the hosts discuss it, because the framing instructed them on pronunciation and the source kept clean.

What quality checks does the pipeline run?

The pipeline enforces ~15 quality rules (all prefixed R-) at the build step. Each rule has an empirical basis — it was added because a past audit found a specific NotebookLM failure mode. Five representative rules:

Rule What it prevents Empirical basis (why this rule exists)
R-PHONETICS-OUT Pronunciation guides leaking into the chapter source (NotebookLM would read them aloud). Audited episode where "Sunnah (SOON-nah)" was vocalized as a parenthetical, breaking flow.
R-HONORIFIC-ONCE Honorific glyphs (ﷺ) repeating multiple times in one chapter (NotebookLM vocalizes each one). Audited episode where "peace and blessings be upon him" was vocalized 9 times in one episode.
R-PRONUNCIATION-IMPERATIVE Pronunciation section in passive list form ("term: respelling") instead of imperative ("Pronounce X as Y"). Audited episode where hosts ignored a passive pronunciation list entirely.
R-WELCOME Awkward "Welcome back to..." openings that NotebookLM tends to default to. Audited episode where the opening felt boilerplate; specific anti-welcome phrasing instructed.
R-NOMETA Hosts breaking the fourth wall ("In this episode we'll be discussing..."). Audited episode where meta-commentary cluttered the substantive content.

Rules tighten over time. When a new NotebookLM failure mode is observed in an audit, a new R-* rule is added. The pipeline runs every rule against every episode. Today's Kitab al-Riyad session has fixed 4 rule-engine bugs (X3, X4, X5, X6) — each was a misalignment between rule intent and rule implementation, not a flaw in the rules themselves.
