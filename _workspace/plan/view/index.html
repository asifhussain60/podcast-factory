7. P0 bug post-mortem

Symptom

Last session: orchestrator ran 19 minutes on Phase 0b, wrote 5 .in.md window files, produced 0 .out.md artifacts. claude -p child was not visible in pgrep.

Root cause

Headless claude -p invocations require explicit permission flags to perform file-write operations. Default behaviour is to refuse the write tool, returning rc=0 with a refusal in stdout. Orchestrator only inspects out_path.exists(), so the refusal is invisible — log shows NO ARTIFACT and proceeds to the next window.

Fix (P0.1 + P0.2)

--- a/scripts/podcast/_chunking.py
+++ b/scripts/podcast/_chunking.py
@@ -190,7 +190,12 @@ def _run_one_window(...):
     _atomic_write(in_path, body)
     proc = subprocess.run(
-        [CLAUDE_CMD, "-p", prompt],
+        [CLAUDE_CMD, "-p", prompt,
+         "--permission-mode", "acceptEdits"],
         timeout=600, capture_output=True, text=True,
     )
-    if not out_path.exists():
+    if proc.returncode != 0 or not out_path.exists() or out_path.stat().st_size == 0:
+        raise WindowFailed(
+            f"win-{idx:03d}: rc={proc.returncode} "
+            f"out_exists={out_path.exists()} stdout={proc.stdout[:200]}"
+        )

P3 makes this fix obsolete because the SDK migration removes the shell-out entirely — Python writes the file itself after receiving the streamed text. Until P3 ships, P0 is the band-aid.

8. Anthropic SDK call shape (P3)

The Python code writes the file itself after receiving the assembled text — eliminating the entire class of "rc=0 but no artifact" bugs. Token-level deltas drive the heartbeat in real time. For interrupted streams, the wrapper retries with a Claude-4.6 continuation request containing the partial response.

10. Dos & Don'ts (curated)

✅ DO

  • Always pass --permission-mode acceptEdits to claude -p with file-write tools P0
  • Verify artifact exists AND is non-empty after every LLM call P0
  • Stream LLM responses with messages.stream() + get_final_message() for any call with max_tokens > 4096 P3
  • Write heartbeat every ≤30 s from a daemon thread, atomic JSON P1
  • Spawn long jobs as subprocesses of the API, never as in-process BackgroundTasks P5
  • Use exponential backoff on Azure DI 429: 2-5-13-34 s (already done)
  • Keep state.json + filesystem authoritative; SQLite is index/cache only P5
  • Append-only checkpoints (.done markers, log rows) — never overwrite

❌ DON'T

  • Don't trust subprocess rc=0 alone — always verify artifacts
  • Don't run orchestrator in FastAPI BackgroundTasks — kills resilience
  • Don't split PDFs until Azure DI 500 MB / 2000-page limit is actually hit (no current book needs it)
  • Don't conflate _workspace/ with content/ — _workspace = tooling/plans, content/ = shipped
  • Don't mix journal and podcast — separate skills, agents, state, services
  • Don't add SQLite before P5 — JSON files suffice for current scale
  • Don't rename phases mid-flight — defer to P6 after end-to-end is stable
  • Don't parallelize per-chapter LLM calls until cost-tracking is in place P7
