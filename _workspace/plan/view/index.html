For a non-technical reader

The shortest possible answer to "what is this?".

What it is

A book-to-podcast pipeline. Feed in a scholarly book (often classical Arabic). Get out a NotebookLM-ready text bundle that turns into a high-quality audio overview.

Why the plan exists

To turn the pipeline from a prototype that works on one book into a durable system that works on seven books, with cost caps, recovery, and quality gates that humans can audit.

What ships when

Wave 1 lands the guardrails. Wave 2 makes the agent observable. Wave 3 proves it on the corpus. Waves 4–6 add control, polish, and deferred bets.

Where the real leverage lives

NotebookLM's voices, cadence, accents, humor, and music are fixed — we cannot pick them. What we can shape is the kind of conversation those fixed hosts have, by handing them podcast-intelligent source architecture instead of raw prose. Most teams upload PDFs and get audiobook output; we engineer bundles with explicit host prompts, episode arc tables, listener-objection blocks, debate framing, segment headers, and "do-not-say" guardrails so the same hosts deliver a different show. See the Steering NotebookLM tab for the full ten-lever toolkit.

How to open these views

All views are static HTML — no server, no build.

From the file manager

Double-click any of the four HTML files. They open in your default browser and resolve assets via relative paths.

view/index.html

From a terminal

Open the landing page directly with the system handler:

open _workspace/plan/view/index.html

Offline by design

No external CDNs are required to render. SVG diagrams are inlined. JavaScript ships locally under assets/js/. This works on a plane, on a closed network, and five years from now when the dependency in question is no longer hosted.
