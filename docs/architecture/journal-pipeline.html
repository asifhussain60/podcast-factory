The five things to remember

·· phase 1 ··

The scratchpad is the workshop

The actual chapter file is sacred. All the rebuilding happens in a separate scratch-<chapter>.txt file that gets deleted after the chapter is finalized. Asif reads it in his editor, not in chat — chat is for section-by-section feedback only.

·· phase 2 ··

Sections, not rewrites

The scratchpad is divided into numbered sections. Asif says "section 5 needs more humour" and only that section is updated, in place. New material inserted between two sections becomes 3a, never renumbering the rest.

·· phase 3 ··

Five quality loops, all silent

Voice, architecture, craft, governance, extraction. They are not separate approval steps — they run automatically around every edit. The only thing Asif sees is the rebuilt section.

·· delta protection ··

His edits are sacred

When Asif edits a chapter file between sessions, the next session detects the change and locks the paragraph. The skill can fix grammar and move it for better flow, but it cannot rewrite his words. Even if it thinks its phrasing is better.

·· phase 4 ··

Nine ceremonial steps to finalize

Seam check, move, lock, update three indices, full compliance scan, snapshot save, delete scratchpad. The list looks long because each step keeps a different invariant true. Skip one and the next session reads stale state.

·· the shared brain ··

Arabic terms defer to the manifest

When a chapter introduces or changes an Arabic translation, the skill consults the cross-skill manifest under content/_shared/arabic/ first. Memoir voice choices that have already shipped win — but the shared file is the default authority.

·· the principle ··

The pipeline is built around one rule: Asif IS Babu. Claude is the assistant that helps him refine and finalize his words. Every guardrail in the diagram above exists to keep Asif's voice intact while still making the writing process faster.
