podcast · pipeline · end-to-end

From any source, any format to two files in NotebookLM

The podcast skill never writes a single line of dialogue. It accepts any source — book, PDF, MP3, Word, PowerPoint, Excel, transcript, plain text — normalizes it to text in one universal intake step, then prepares a perfectly steered packet of source material so that NotebookLM's two hosts say exactly the right things on their own. This page shows every file, every agent, every challenger pass, and every script that gets us there.

·· the whole journey ··

One book, one full series

Read top to bottom. The path forks early — any long-form source (PDF · audio · Word · PPT · Excel · book) triggers a full Phase 0 ingestion; a memoir chapter takes the Extract short-cut. They reunite at Phase 1.

·· moment by moment ··

The seven things to remember

·· session start ··

Five shared files before anything book-specific

The pronunciation brain at content/_shared/arabic/ is read first, every run. That is what guarantees an Arabic term in book A is pronounced the same way in book B six months later.

·· phase 0 ··

A long source is a series, never an episode

A 30-page PDF, a two-hour audio recording, a 40-slide deck — each becomes 5-to-10 designed chapters, each one its own NotebookLM upload. Phase 0a normalizes any format to text first; Phase 0d re-segments by meaningful thematic units; Phase 0e enriches each chapter with authoritative material from seven tradition-tiers.

·· extract mode ··

Memoir chapters take the short-cut

When the source is a memoir chapter, all of Phase 0 is skipped. The chapter file is read directly via the one sanctioned door between the memoir room and the podcast room, governed by a YAML contract per chapter.

·· the chapter file ··

One chapter, one episode

Strict 1:1 mapping enforced by the build script. Each chNN-<slug>.txt is the source of exactly one EP##-<slug>.txt. The slug is identical on both sides. Skip the chapter, and there is no episode to build.

·· phase 4 ··

The challenger agent must say SHIP-READY

A separate agent (podcast-challenger) reads every chapter and framing literally, runs 30+ checks across seven categories, auto-fixes the deterministic problems, and writes a verdict. The bundle is not ready until that verdict is green.

·· two-file model ··

Why two files, not one

If you uploaded a single concatenated file, NotebookLM would treat the customize prompt as source content and the hosts would read it out loud. Two files, two folders, zero ambiguity.

·· refinement ··

The scratchpad is the dialogue surface

After the episode is built, the chapter is mirrored into a scratchpad with an @@ marker legend prepended. Asif marks the scratchpad up. The skill applies the markers, rewrites the chapter, and re-runs the build script.

·· what this isn't ··

NOT a podcast generator

This skill does not write scripts, does not generate audio, does not automate NotebookLM, and does not publish episodes. NotebookLM does those things. The skill steers NotebookLM.

·· the test ··

A strong bundle is one where, after NotebookLM finishes the Audio Overview, the listener thinks "the hosts said exactly what mattered." If the hosts drift or ramble, the bundle was weak — fix the bundle, not the model.

·· architecture v3.5 · empirical feedback loop · 2026-05-17 ··

What happens AFTER upload · the audit_transcript loop

The pipeline above ends at "upload + click Generate". The v3.5 architecture adds a second loop that runs after NotebookLM produces the audio overview: transcribe it, drop into transcripts/, run audit_transcript.py, get a per-episode regression report. The next chapter/framing revision pass is informed by what the hosts actually said, not by what we hoped they would say.
·· step 1 ··

Transcribe the audio overview

Via Turboscribe (or equivalent). Save to content/podcast/<book-slug>/transcripts/EP##-<slug>.transcript.txt. Slug must match the chapter + episode (1:1:1 mapping; the audit script auto-resolves the path from the EP id).

·· step 2 ··

Run the audit

python3 scripts/podcast/audit_transcript.py BOOK_DIR EP##-<slug> writes _system/audit-EP##-<slug>.md. The report measures phonetic doublings, mangled names, modernization injections, surprise-noise loops, welcome violations, honorific repetition, abbreviation hits, and filler density.

·· step 3 ··

Use the report as the regression target

The next pass through Phases 3–4 (rewrite chapter, rewrite framing, re-run build_episode_txt.py) is informed by the audit. Did Sahih Sitta still get doubled? Open the framing's ## Pronunciation block — add an explicit imperative directive. Did "deep dive" appear? Extend the ## Do not DENY block.

·· step 4 ··

Re-upload · re-transcribe · re-audit

Each pass closes the gap. The first pre-v3.5 audit established the baseline; every subsequent audit is measured against the prior report. Goal: drive phonetic doublings to zero, mangled names to zero, modernization density below 0.5 per 1k words, surprise density below 1.0 per 1k words.

·· why this loop closes the architecture ··

Before v3.5, the pipeline was open-loop. We wrote chapter + customize prompt → uploaded → trusted NotebookLM produced what we steered for. There was no sensor at the output. Empirically that meant: 5 episodes of Ayyuhal Walad shipped with systematic phonetic doublings, mangled names, surprise loops, and modernization injections that nobody had measured. Architecture v3.5 closes the loop: sensor (transcribe), analyser (audit_transcript.py), actuator (the producer + challenger), and a target metric per failure mode.
