How the podcast factory actually works

A scholarly Arabic book goes in. A NotebookLM-ready coordinated source bundle for every chapter comes out. Between those two points sit nine deterministic phases, a per-chapter quality loop with hard stop-on-failure, seven gates before anything publishes, and a small constellation of agents that drive each step. This page is the single architectural view — scroll through, hover and click the diagrams, and you should be able to brief a skeptical VP over coffee.

Section 1

What this repo is

One paragraph, no diagram — the hook.

podcast-factory is the Python + Claude + Azure pipeline that turns scholarly Arabic books into NotebookLM podcast series. A PDF lands in content/drafts/<slug>/ and walks through a fixed sequence: Azure does OCR and Arabic-to-English translation; Claude does English refinement, phonetic markup for Arabic terms, chapter design, and enrichment with approved-source quotations; then a per-chapter convergence loop runs the chapter through a challenger agent (a quality reviewer that emits P0 / P1 / P2 findings, where P0 is ship-blocking) and a fixer until the chapter is either SHIP-READY or it hits a hard stop. When all chapters clear the loop and seven publish gates pass, the artifacts move from content/drafts/ to content/published/. The repo is machine-agnostic (most work is remote Claude + Azure calls), runs from a single working branch (develop), and uses a per-content typed branch (book/, doc/, lecture/, etc.) to isolate each piece of work until it publishes. The memoir engine and static site moved to the sibling journal repo on 2026-05-22 and are not part of this view.

Section 2

Content lifecycle

From a PDF on disk to a publishable bundle.

Content moves through three regions on disk. Sources live anywhere the operator drops them (typically _workspace/Books/). Workshop is content/drafts/<slug>/ — every script in the pipeline reads and writes here. Published is content/published/books/<slug>/ — populated only by publish_to_library.py via the podcast-publisher agent, never edited by hand. The published tree is filesystem-only and intentionally not git-tracked, because it is a derived artifact — deletable and re-generatable from the workshop at any time.

A PDF (left) is copied into the workshop, walks the pipeline in-place, then a one-way promotion lands the bundle in the published catalog.
Section 3

Pipeline phases

The deterministic sequence the orchestrator drives.

The orchestrate_book.py driver runs phases in order, each a discrete checkpoint that writes a named artifact under the workshop's _system/ directory and updates orchestrator-state.json atomically. Phase 0a is ingest (Azure OCR + Arabic-to-English translation). 0b through 0e are LLM-authored passes (refinement, phonetic markup, chapter design, enrichment). 0f is the single human halt — the orchestrator writes series-plan.md and stops; the operator reviews it and runs --resume to continue. On resume, the per-chapter convergence loop runs (see Section 4), then 0g registers the series, the publisher publishes, the trainer learns from findings, and the content branch merges back to develop.

Section 4

Per-chapter convergence loop

The quality gate that runs on every chapter before publish.

After Phase 0f human review, each chapter walks an isolated convergence pass. The challenger (a Claude-driven semantic reviewer at infra/claude-agents/podcast-challenger.md) reads the chapter and emits a verdict — SHIP-READY, SHIP-WITH-CAUTION, or BLOCKED — alongside finding counts by severity (P0 blocks ship, P1 is an advisory, P2 is informational). On any non-SHIP-READY verdict, a fixer Claude call edits the chapter to address the listed findings and the loop runs again. The loop is capped at three outer iterations. SHIP-WITH-CAUTION is only accepted at iteration 2 or later (so a first-pass caution still triggers a fix-and-retry). If iteration 3 still has unresolved P0s, the loop HALTS as FAILED and surfaces to the orchestrator — there is no silent downgrade. The cap constants live in scripts/podcast/_convergence.py: MAX_OUTER_ITERATIONS = 3, MAX_FIXER_ATTEMPTS_PER_P0 = 3, SHIP_WITH_CAUTION_MIN_ITER = 2.

Section 5

Branch policy

One typed branch per piece of content, off develop.

Every new piece of content gets its own branch off develop. The prefix is decided by the content's category field through scripts/podcast/_branching.py's branch_name(category, slug) — never hardcode book/<slug> anywhere else. The branch is cut at intake (intake_book.py), the entire pipeline runs on it, and only after the publish step completes does it merge back to develop with --no-ff. This isolates in-flight work, gives a clean per-content ledger, and lets several books be in flight at once without cross-contamination. Slugs are always the full kebab-cased name (never book/KaR — only book/kitab-al-riyad).

Section 6

Publish gates G1 to G7

Seven sequential checks; any failure aborts the publish.

scripts/podcast/publish_to_library.py runs the gates strictly in order. Every gate must return OK or the script exits with code 1 and writes nothing to the published catalog. G1 through G3 check structure on disk (chapters and episodes exist, pair up, are sequentially numbered). G4 runs build_episode_txt.py on every episode and requires zero P0 flags. G5 reads the orchestrator state file and only proceeds if the pipeline is in a shippable phase. G6 confirms the target directory is safe to wipe-and-recreate. G7 refuses to publish anything that did not pass a real challenger convergence pass — closing the historical bypass where non-orchestrated books could reach the audience without the quality loop.

Section 7

Category T — doctrinal accuracy

Five checks that protect the religious-tradition integrity of every chapter.

Category T checks (T1 through T5, implemented in scripts/podcast/_doctrinal.py) catch doctrinal slips that an English-trained LLM tends to make on Ismaili Shia material — notably the phrase "Imam Ali", which is the Twelver Shia convention but doctrinally incorrect here (Ali ibn Abi Talib is the Father of the Imams, not the first Imam). The data lives in content/_shared/islam/imam-lineage-ismaili.yml (the canonical seven-Imam Ismaili lineage and the forbidden-title list) and content/_shared/islam/naming-conventions.yml (the forbidden-phrase scrub). The checks are wired into build_episode_txt.py as a hard gate, so a chapter containing "Imam Ali" cannot ship. T4 and T5 are reserved rule IDs — T4 needs a Farman dataset that does not exist yet, T5 scans for weak or fabricated hadith and is currently a stub awaiting the trainer to populate patterns.

Decision tree: a chapter enters at top, splits across the five Category-T checks, and exits as a P0 (ship-blocking), P1 (advisory), or pass. The two reserved-but-stubbed checks (T4, T5) sit in gray.
Section 8

Agent surface

The ten agents under infra/claude-agents/, how they compose.

Each agent lives as a markdown spec under infra/claude-agents/ with YAML frontmatter declaring its triggers and tool permissions. The podcast-orchestrator sits at the center — it drives the full PDF-to-merged-branch flow by invoking the other agents at the right step. The challengers (one for audio, one for slide decks) judge quality. The publisher moves shipped content to the catalog. The trainer closes the loop by mining cross-book findings into spec edits. The auditor and blueprint agents are support: the auditor checks repo-level health, blueprint classifies a book's genre and proposes an episode plan. refine-prompt and docs-updater are operator utilities — the first turns raw notes into a Claude Code instruction-paragraph, the second regenerated this very page.

Section 9

Azure stack

Four Cognitive Services plus storage. Phase-by-phase wiring.

Azure runs the work that does not belong in a Python or Claude process: OCR of scanned Arabic PDFs (Document Intelligence), Arabic-to-English translation (Translator), named-entity recognition on the refined English (Language / TextAnalytics), and optionally Urdu / Arabic speech-to-text for lecture content (Speech, behind feature flag). Credentials resolve through environment variables first (for CI) and fall back to macOS Keychain entries under the azure-journal-* naming convention — see _workspace/setup/azure-stack.md for the full keychain map and the recreate-from-scratch procedure. Resource group is rg-journal-ai (kept from the pre-rename journal repo); the resources themselves were never renamed because the rename was a config-label change only.
