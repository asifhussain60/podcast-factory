The podcast-factory pipeline

A long-scrolling tour of how a scholarly Arabic PDF becomes a two-host NotebookLM audio overview — the rooms it moves through, the phases that edit it, the gates that bless it, and the agents that drive the whole thing.

01 — THE HOOK

What this repo is

One paragraph. No diagram. The shape of the whole thing.

podcast-factory is the pipeline and reader app for turning scholarly Arabic books (Ismaili tradition, primarily) into two-host audio overviews via Google's NotebookLM. Source PDFs land in a per-book workshop under content/drafts/<slug>/; the pipeline runs a sequence of phases — Azure OCR + translation, English refinement, phonetic extraction, chapter design, enrichment, then per-chapter authoring under a challenger convergence loop — and shipped output moves to content/published/books/<slug>/ after passing seven named gates. A small set of Claude agents (orchestrator, challenger, extract, blueprint, publisher, trainer, auditor, slide-deck-challenger) authors, validates, and ships. The repo is machine-agnostic since 2026-05-23 — one repo, one working branch (develop), promotion to main only on explicit operator approval.

The podcast-reader Astro app under podcast-reader/ reads from content/drafts/<slug>/chapter-contracts/ and lets the operator annotate chapter contracts in the browser before they go to NotebookLM. The memoir engine ("What I Wish Babu Taught Me") moved to a sibling repo at ~/PROJECTS/journal/ on 2026-05-22 — that is a fully disconnected concern. This page is about the pipeline only.

02 — CONTENT LIFECYCLE

From PDF on disk to published audio bundle

Three rooms — Source, Workshop, Published. Two scripts move material between them.

Every piece of content lives in one of three places at any time. The Source is the raw PDF (or DOCX, or audio file) somebody hands us — it sits anywhere on disk, untouched by the pipeline. The Workshop at content/drafts/<slug>/ is where the pipeline reads and writes — every intermediate artifact, every chapter contract, every challenger report. The Published tree at content/published/books/<slug>/ is the audience-facing catalog and is written only by publish_to_library.py after gates G1–G7 pass.

03 — PIPELINE PHASES

Eleven phases from PDF to shipped audio bundle

Click any phase to see what it reads, what it writes, and how it fails.

The orchestrator at scripts/podcast/orchestrate_book.py drives a book through eleven phases — first six are book-level (run once over the whole PDF), the last few are per-chapter (run N times, once per extracted chapter). There is exactly one human gate, at Phase 0f, where the operator reviews the chapter list and either authorizes the rest of the run or fixes the design before continuing.

0a

Azure OCR + Translation

PDF goes to Azure Document Intelligence for OCR; the Arabic text is then translated to English via Azure Translator. Output is a raw bilingual artifact.

Reads
The source PDF dropped at the operator-supplied path.
Writes
_system/source/raw-ar.txt + raw-en.txt in the workshop.
Cost
Azure metered — low single-digit dollars per typical 200-page book.
Fails when
Azure credentials missing (run store-keychain-keys.sh); Translator transient — retry with --retry-phase 0a.

04 — CONVERGENCE LOOP

How one chapter passes (or fails) quality review

3 outer iterations × 5 inner challenger iterations = 15 passes maximum per chapter, then HALT.

Every chapter goes through the same convergence loop: challenger reads the chapter + framing + episode and renders a verdict — SHIP-READY, SHIP-WITH-CAUTION, or BLOCKED. SHIP-READY ships. SHIP-WITH-CAUTION ships only at iter ≥ 2 (one round of P1 fixer first). BLOCKED triggers the fixer agent on P0 findings, then re-runs challenger. After 3 outer iterations with anything unresolved, the loop halts the orchestrator and surfaces the failure — no silent downgrades to ship.

Read top-to-bottom · arrows are state transitions
SHIP-READY · ships immediately SHIP-WITH-CAUTION · ships at iter ≥ 2 BLOCKED · fixer attempts, then retry

The dashed arrows are the retry loop. At outer iteration 3, any unresolved verdict halts the orchestrator. Prior behavior silently downgraded BLOCKED to FORCE-SHIP-CAUTION; that bypass was closed alongside publish gate G7.

05 — BRANCH POLICY

One typed branch per piece of content

Single source of truth: scripts/podcast/_branching.py.

Every new piece of content is processed on its own typed branch off develop. The branch is created at intake time and merged back to develop only after publish completes. The prefix is determined by the content's category field. Slugs are always the full kebab-cased name — never abbreviate (so book/kitab-al-riyad, never book/KaR).

Branch prefix Category Example branch name
book/ books book/kitab-al-riyad
doc/ documents doc/imamat-overview-2026
lecture/ lectures lecture/kunooz-al-hikmah-session-04
article/ articles article/the-meaning-of-tariqat
letter/ letters letter/al-aziz-billah-to-the-jamat
interview/ interviews interview/qazi-noman-on-tawil
draft/ (fallback — category unknown) draft/pending-classification

The draft/ prefix is the fallback when category is missing or unrecognized — it keeps the branch trackable while signaling that classification is pending. Promotion to develop happens only after publish; promotion to main needs explicit operator approval (GitHub ruleset enforces).

06 — PUBLISH GATES

Seven gates that must pass before content ships

Click any gate to see what it checks, what failure means, and how to recover.

scripts/podcast/publish_to_library.py runs seven gates in order. Any failure aborts the publish — nothing reaches content/published/books/<slug>/ until every gate is green. G7 (challenger convergence) was added 2026-05-24 to close the bypass that previously let non-orchestrated books reach the audience without ever clearing the convergence loop.

G1

Required structure

chapters/*.txt and episodes/*.txt both exist and are non-empty.

Why: No point doing anything else if the raw artifacts aren't on disk.

FAIL means a phase didn't run or wrote nothing. Re-run the relevant phase via --retry-phase.
▶
G2

Pair completeness

Every EP##-<slug>.txt has a matching ch##-<slug>.txt, and vice versa.

Why: Episodes and chapters travel as pairs to NotebookLM; an orphan on either side means the bundle is incomplete.

FAIL usually points to a manual edit gone wrong; check the orchestrator-state.json for the chapter index.
▶
G3

Sequential numbering

Chapter + episode numbers are purely sequential 1..N — no letter suffixes (ch14a), no decimals (ch3.5).

Why: The reader app + catalog assume contiguous numbering. Mirrors the future S-EPISODE-SEQUENTIAL challenger check, inlined here.

FAIL means a phase 0d rerun left a gap; reset and re-run via --retry-phase 0d.
▶
G4

Build-clean (P0=0)

Running build_episode_txt.py on every episode returns zero P0 flags. P1 advisories warn-only by default; --strict elevates P1 to blocking.

Why: Even after challenger convergence, a doctrinal P0 (forbidden phrase, lineage swap) must not slip past the static builder.

FAIL means an "Imam Ali"-class phrase leaked back in. Run the doctrinal checks from Section 7 to localize.
▶
G5

State checkpoint

orchestrator-state.json shows phase=done OR phase=per-chapter with a shippable phase_status.

Why: Don't publish a book the orchestrator thinks is still in flight; the state.json is the source of truth.

FAIL usually means an unclean shutdown left phase_status=running (the known orchestrator-resume bug). Bypass with --force only after manual review.
▶
G6

Library-target sane

Target content/published/books/<slug>/ either doesn't exist (we create) or --overwrite (default) wipes and recreates. --no-wipe coexists.

Why: Don't blow away a target that turns out to be a symlink or outside the library tree.

FAIL when target is a symlink (refuses wipe) — investigate before continuing.
▶
G7

Challenger convergence

Book's pipeline_mode is NOT in the non-converged set, AND challenger-report.md verdict is in {SHIP-READY, SHIP-WITH-CAUTION}.

Why: Closes the bypass where non-orchestrated books or N/A reports reached the audience without ever clearing convergence. Sibling closure to the silent-downgrade fix in _convergence.py.

FAIL means either pipeline_mode=non_orchestrated_mode_2 or verdict missing. Use --allow-mode-2 only after manual challenger pass.
▶
07 — DOCTRINAL ACCURACY

Category T — five checks for Ismaili-tradition correctness

Each chapter walks through T1 → T5; P0 findings block, P1 findings flag.

The _doctrinal.py module runs five checks against every chapter + framing + episode. Data backing the checks lives in content/_shared/islam/ (imam lineage, naming conventions, canonical attributions). T4 and T5 are stubs reserved for future datasets — the rule-id taxonomy stays complete so the trainer can wire them later without a renumbering.

Note on lineage: in this tradition, Ali ibn Abi Talib is the Father of the Imams, not the first Imam. The first Imam is Hassan. The phrase "Imam Ali" is therefore P0 forbidden — it reflects the Twelver convention, not the framing of these books.

A chapter enters at the top · cleared chapters exit at the bottom-left

The decision tree runs T1 → T5 in sequence on every chapter. A clean walk (no findings) ships left; any P0 ships right into the convergence loop's BLOCKED branch. The seven valid Imams in the lineage are Hassan, Hussain, Ali Zayn al-Abidin, Muhammad al-Baqir, Ja'far al-Sadiq, Isma'il ibn Ja'far, and Muhammad ibn Isma'il.

08 — AGENT SURFACE

Ten Claude agents, one orchestrator at the center

Orchestrator is the only agent that drives the others; the rest are leaves.

The pipeline ships with ten Claude agents under infra/claude-agents/. podcast-orchestrator sits at the center — it drives every other agent at the appropriate phase. The refine-prompt and docs-updater agents are tooling agents not invoked by the orchestrator; they live on the edge of the graph. Eight agents radiate from podcast-orchestrator. The refine-prompt agent (top-left) is operator tooling and lives off the invocation graph. The docs-updater agent that regenerated this very page is also operator tooling — it reads the repo, never modifies pipeline state.

09 — AZURE STACK

Four Azure resources, one phase per resource

All keys in macOS Keychain; infra/azure/azure-config.env is tracked non-secret config.

The pipeline uses four Azure Cognitive Services resources for OCR, translation, and named-entity recognition. Resource group rg-journal-ai in region eastus. Secrets live in macOS Keychain (one entry per field — endpoint, key, region), populated by infra/azure/store-keychain-keys.sh. _azure.py resolves credentials in order: env var first (for CI), then Keychain (local Mac). Speech is feature-flagged on (currently used for Urdu lecture transcription on the kunooz-al-hikmah project).

Pipeline phases call out left → right to Azure services

Phase 0a fans out to both Doc Intelligence and Translator. Phase 0a.5 (NER pre-seed) is wired but framework integration is pending. Speech is opt-in via ENABLE_SPEECH=true in azure-config.env. All edges are authenticated by keys resolved through the Keychain.
