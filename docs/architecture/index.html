SINGLE-VIEW ARCHITECTURE

How podcast-factory turns scholarly books into two-host audio.

A 700-page Arabic theological treatise goes in. NotebookLM-ready chapter bundles, plus a published two-host audio overview, come out. The path between is a multi-phase pipeline with named safety gates, typed branches per piece of content, a per-chapter convergence loop with hard caps, and a small council of Claude agents that author, critique, and ship. This page shows every moving part. Read it with coffee.

01 — THE HOOK

What this repo is

One paragraph. No diagram. The shape of the whole thing.

podcast-factory is the pipeline + reader for converting scholarly Arabic books (Ismaili tradition, primarily) into two-host audio overviews via Google's NotebookLM. Source PDFs land in a per-book workshop under content/drafts/<slug>/; the pipeline runs a sequence of phases (Azure OCR + translation, English refinement, phonetic extraction, chapter design, enrichment, per-chapter authoring, challenger convergence), and shipped output moves to content/published/books/<slug>/ after passing seven named gates. A small set of Claude agents (orchestrator, challenger, extract, blueprint, publisher, trainer, auditor) authors, validates, and ships. The whole thing is machine-agnostic since 2026-05-23 — one repo, one working branch (develop), promotion to main only on explicit operator approval.

The podcast-reader Astro app under podcast-reader/ reads from content/drafts/<slug>/chapter-contracts/ and lets Asif annotate chapter contracts in the browser before they go to NotebookLM. The memoir engine ("What I Wish Babu Taught Me") moved to a sibling repo at ~/PROJECTS/journal/ on 2026-05-22 — that's a fully disconnected concern. This page is about the pipeline only.

02 — CONTENT LIFECYCLE

From PDF on disk to published audio bundle

Three rooms: Source, Workshop, Published. Scripts move material between them.

Every piece of content lives in one of three places at any time. The Source is the raw PDF (or DOCX, or audio file) somebody hands us. The Workshop at content/drafts/<slug>/ is where the pipeline reads + writes — every intermediate artifact, every chapter contract, every challenger report. The Published tree at content/published/books/<slug>/ is the audience-facing catalog and is written only by publish_to_library.py after gates G1–G7 pass.

Read left to right. A source PDF is copied into the workshop by intake_source.py; once every gate in section 06 passes, the publish_to_library.py script promotes the artifacts to the audience-facing catalog. The PDF itself is never modified.
03 — PIPELINE PHASES

Eleven phases, run by the orchestrator

A "phase" is a single deterministic step the orchestrator advances through.

orchestrate_book.py drives every book end-to-end. Each phase reads from content/drafts/<slug>/, writes its output back into the same workshop, and stamps progress into _system/orchestrator-state.json. The orchestrator halts at Phase 0f so the operator can review the chapter plan before per-chapter authoring burns tokens. After approval, --resume picks up at per-chapter and runs to completion (or hits a convergence cap, see §04).

04 — PER-CHAPTER CONVERGENCE

How a chapter clears (or doesn't)

Three outer iterations, five inner. Hard caps. No silent ships.

Every chapter, after framing + episode-text are built, enters the convergence loop in _convergence.py. The loop runs the challenger agent (which reads the chapter, framing, and episode-text and emits a verdict report), then either accepts the verdict, invokes the fixer agent, or halts. It has three layers of cap: inner (challenger's own 5-iteration cap, enforced inside the challenger), middle (3 outer iterations here, in converge_chapter), and outer (24h/$50 per book, enforced by the orchestrator).

The key rule, locked after a 2026-05-23 audit: BLOCKED verdicts at the iteration-3 cap are NEVER silently downgraded to a ship-state. The chapter is marked FAILED and surfaced to the operator with full P0/P1 carry-over.

The three verdicts. SHIP-READY (green, right) breaks the loop cleanly. SHIP-WITH-CAUTION (gold, down) ships only if it survives at least two iterations. BLOCKED (red, left) invokes the fixer on P0 findings and retries; if iteration 3 still fails, the chapter is FAILED and surfaced — never silently shipped.
05 — BRANCH POLICY

Typed branch per piece of content

One branch per content category, off develop, merged after publish.

Every new piece of content is processed on its own typed branch off develop. The prefix encodes the content category — books go on book/<slug>, articles on article/<slug>, etc. Branches are created at intake and merged back to develop only after publish completes. Slugs are always the full kebab-cased name, never abbreviated (book/kitab-al-riyad, not book/KaR). Single source of truth lives in scripts/podcast/_branching.py.

06 — PUBLISH GATES

Seven gates, run in order, by publish_to_library.py

Pass all seven → promotion to content/published/. Fail one → halt.

publish_to_library.py is the only script authorized to write into content/published/books/. Before it does, it runs seven gates in strict order. Each gate is a small function in the same file (gate_g1_structure, gate_g2_pairs, …). All seven must pass; the first failure stops the publish and explains how to fix.

07 — DOCTRINAL ACCURACY

Category T — the five doctrinal checks

Stops the pipeline from shipping content that says "Imam Ali" or pairs the wrong Imam with the wrong ordinal.

Category T (the "T" is for "tradition / theological") is a set of five rules in scripts/podcast/_doctrinal.py that run on every chapter during the convergence loop. They guard against doctrinal errors that no static linter catches — phrases like "Imam Ali" (forbidden in the Ismaili tradition because Ali ibn Abi Talib is the father of the Imams, not counted in the lineage), or "the 5th Imam Hassan" (wrong — Hassan is #1; #5 is Ja'far al-Sadiq). T1 + T2 + T3 are live and aggressive; T4 + T5 are reserved slots with stub implementations.

One chapter, five checks, two outcomes. The chapter text fans out to five doctrinal checks; their findings aggregate (P0 first, then by check ID). Any P0 → the chapter goes to the fixer. All-clean → on to the rest of challenger's non-T checks. T1–T3 are live now; T4 (Farman date/location) and T5 (weak hadith) are reserved slots with stub implementations awaiting dataset population.
08 — AGENT SURFACE

The agents under infra/claude-agents/

A small council: orchestrator at the centre, specialists arranged around.

Ten agent specs live in infra/claude-agents/. They aren't background services — each is a Claude prompt + tool list that the human (or another agent) invokes at a particular phase. podcast-orchestrator is the only one with a full driver script (orchestrate_book.py); the rest are invoked from the orchestrator or directly by the operator.

09 — AZURE STACK

Four Azure resources, two phases

Document Intelligence, Translator, Language, Speech (optional).

Azure handles the OCR + translation work in Phase 0a (and optional Speech transcription for audio lectures). All four resources live in the rg-journal-ai resource group, region eastus, under the Journal AI — primary subscription. Keys live in macOS Keychain; scripts/podcast/_azure.py resolves them per-call. The fifth slot (journal-speech) is feature-flagged and only active for audio sources like kunooz-al-hikmah.

Phases (left) → Azure resources (right). Phase 0a hits Doc Intelligence then Translator. Phase 0a.5 hits the Language Cognitive Service for named-entity pre-seed (used by the chapter-design phase). The Speech resource is feature-flagged and only fires when ENABLE_SPEECH=true in infra/azure/azure-config.env.
