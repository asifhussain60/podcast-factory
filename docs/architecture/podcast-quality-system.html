podcast · quality system

Seven silent loops, a seven-tier pyramid, a challenger and a trainer · the loop closes itself

Quality control runs on five layers: seven silent loops every bundle passes through during authoring; a strictly tiered list of where enrichment is allowed to come from; an independent challenger agent with a growing set of check categories (A–K, M, N, O, Q) that keeps iterating until the bundle is genuinely ready; an empirical feedback loop that audits the actual NotebookLM-generated transcript against every directive the customize prompt was supposed to enforce; and — landed 2026-05-18 — a closed-loop learning substrate at content/podcast/.skill/_learning/ where every challenger and audit finding emits into a single append-only JSONL ledger, recurring patterns become proposals, and a podcast-trainer agent promotes rule changes onto the book branch only when a regression gate against seven frozen fixtures stays green. The challenger reads two normative rule files as its authority: notebooklm-source-chapter-rules.md (17 chapter contracts) and notebooklm-customize-prompt-rules.md (12 framing contracts including welcome opening, anti-repetition, no-irrelevant-background, name aliasing, interruption avoidance, imperative-pronunciation, modernization DENY, surprise-noise DENY, and the no-read-aloud guard).

·· 2026-05-17 · architecture v3.5 ··

After a 5-transcript audit of Ayyuhal Walad exposed systematic failure modes (phonetic doublings like "Sahih Sitta, sahasita"; mangled names like "tassel wolf" for Tasawwuf; "the EI" for Ihya Ulum al-Din; 9 expanded honorifics in one episode; >40 surprise-noise injections across 5 episodes), the architecture pivoted: phonetic guidance moved from inline chapter parens (which NotebookLM reads aloud as content) to imperative customize-prompt directives (which the voice model uses silently). Three new chapter rules (R-PHONETICS-OUT, R-HONORIFIC-ONCE, R-NO-ABBREVIATION) and four new framing rules (R-PRONUNCIATION-IMPERATIVE, R-NOMODERNIZE, R-NOSURPRISE, R-NO-READ-PROMPT) make this enforceable. Three new challenger categories (M, N, O) and a new audit_transcript.py script close the empirical feedback loop.

·· four new loops · normative rules ··

The customize-prompt enforcement layer

Loops H, I, J, K each enforce a rule from notebooklm-customize-prompt-rules.md. The rule file is the single source of truth — both the producing skill and the challenger read it. No drift possible.
·· loop H · welcome + landing ··

Open with a welcome and summary; close on a question

The customize prompt must direct hosts to open with a brief welcome and a 2–3 sentence summary naming source + tension + landing question. Forbids "today we'll discuss" boilerplate. Closes on the unresolved tension or a single sharp line — never a recap of what was just said. Rules R-WELCOME + R-SUMMARYTAIL.

·· loop I · anti-repetition + background ··

Land each beat once; no Wikipedia detours

The customize prompt must forbid restating the central thesis more than twice, re-citing quotes, or summarizing what was just said. Biographical / historical background about author or translator appears only when pertinent and only once — never opening the episode, never returned to. Rules R-NOREPEAT + R-NOBACKGROUND.

·· loop J · name aliasing ··

Long name once, alias thereafter

"Imam Abu Hamid Muhammad al-Ghazali" said in full once, then "Ghazali" for every subsequent reference. "Hatim bin Ism al-Asamm" → "Haatim". ~30 canonical aliases live in content/_shared/arabic/05-name-alias-policy.md; both chapter and framing apply them. Rules R-NAMEALIAS + R-NAMES.

·· loop K · interruption avoidance ··

Each host completes a thought

The customize prompt must forbid mid-sentence interjections ("yeah", "right", "exactly") and talking-over. Hosts may pick up the thread after the other lands a beat, but not before. Concrete filler-word vocabulary is named in the framing so NotebookLM's voice model has a list to suppress. Rule R-NOINTERRUPT.

·· why externalize ··

Before the rule files existed, every rule had to be written twice — once in SKILL.md (so the producer obeys it during authoring) and once in the challenger (so the reviewer catches violations). Two copies = drift risk = silent rule-fork. Now both producer and reviewer read the same file. Add a new rule by adding a new section to the rule file; both pick it up next run.

·· three new categories · empirical-audit-driven ··

Categories M, N, O · added 2026-05-17

After the Ayyuhal Walad 5-transcript audit, three new check categories were added to the challenger. Each enforces a rule the soft-prose tone constraints could not.
·· category M · empirical-transcript loop ··

Modernization + surprise audit

The only challenger category that reads the actual NotebookLM transcript. Verifies the framing carries the canonical DENY blocks for modernization (Twitter, X, social media, algorithm, content creator, deep dive) and surprise noise ("wow", "it's chilling", "right?", "exactly"). Counts injections in the transcript and reports density per 1,000 words. Rules: R-NOMODERNIZE + R-NOSURPRISE.

·· category N · phonetic-as-content audit ··

Inline phonetics out → customize prompt in

Chapter contains zero inline (PHO-NE-TIC) parens (auto-fix stripped). Framing's ## Pronunciation block uses imperative form (auto-fix converts legacy passive lists). Every chapter Arabic term has a matching Pronounce "..." line in the framing. Framing ends with the no-read-aloud guard. Transcript scanned for doublings ("Sahih Sitta, sahasita") which signal residual inline phonetics. Rules: R-PHONETICS-OUT + R-PRONUNCIATION-IMPERATIVE + R-NO-READ-PROMPT.

·· category O · honorific + abbreviation audit ··

Honorific once · titles in full

Each honorific phrase form ("peace and blessings be upon him", "(PBUH)", "(AS)", "(RA)", "ﷺ") expanded at most once per chapter (auto-fix strips 2nd+ occurrences). No abbreviated work titles ("the Ihya", "EI", "the Nahj", "Sahihayn") — auto-fix replaces from the canonical map. Transcript scanned for the same patterns to confirm the chapter-level fixes propagated. Rules: R-HONORIFIC-ONCE + R-NO-ABBREVIATION.

·· category Q · cross-book + chapter-set design · added 2026-05-18 ··

Category Q · cross-book bleed and series balance

When the repo hosts multiple podcasted books, Q checks that terms, framings, and phonetic guidance from one book do not bleed into another. It also checks the chapter-set as a whole: word-count variance, title uniqueness across the series, contract coverage, lineage integrity.
·· Q1 · pronunciation manifest scope ··

Each book owns its phonetics

Per-book _system/pronunciation.md entries must not silently shadow another book's. Shared canonical terms inherit from content/_shared/arabic/; per-book overrides stay scoped. Flag if a term from book A appears in book B's manifest without independent justification.

·· Q2 · chapter-set balance ··

~30% word-count variance across the series

A series with 1,200-word and 4,000-word chapters has the wrong shape. Q reads every chapter file in chapters/, computes the variance, and flags series that exceed the 30% band. Re-segmentation, not patchwork.

·· Q3 · chapter-contract coverage ··

Every chapter has a YAML contract

Every chapters/chNN-<slug>.txt must have a paired chapter-contracts/<slug>.yml (title · content-type · enrichment scope · anchor passages). Missing contracts halt the run; orphan contracts flag for cleanup. Lineage validator runs on every full extract.

·· producer rules · added 2026-05-18 ··

Conversation choreography + R-NOMODERNIZE softening

Two rule additions that emerged from re-listening to the post-v3.5 Ayyuhal Walad episodes. Both live in two-host-framing.md and notebooklm-customize-prompt-rules.md; both producer and challenger read them.
·· R-CHOREOGRAPHY ··

Driver · Color · one cross-introduction beat

Host A is the Driver (owns intro, transitions, pacing, resets between beats, landing) and the Curious Mind (listener stand-in). Host B is the Color (analytical contrast, related works, the counter-reading) and the Scholar. Each episode plants at least one beat where one host introduces a passage, citation, or question from the source that the other has not led toward. Breaks mechanical Q-and-A rhythm.

·· R-NOMODERNIZE · softened ··

Platform DENY stays · modern-life analogies DO

The platform/anachronism DENY list is unchanged (Twitter, X, algorithm, deep dive, content creator, dopamine hit, …). What was added is the explicit positive form: hosts DO use generic modern-life situations as comprehension aids — a parent worrying about a child, an employee resenting a coworker, learning a craft over years. Each analogy lands in a single beat and returns to the source.

·· pending · Arabic TTS protocol · Track A approved · awaiting implement ··

The tracked protocol at content/podcast/.skill/handbook/arabic-tts-protocol.md (cold-start item 17) proposes four refinements that have NOT yet landed in the rules files: (1) ## Pronunciation renamed to ## Phonetic Key (TTS Pronunciation); (2) R-PRONUNCIATION-IMPERATIVE splits into Conversational (smooth Anglicized — Inshallah → in-shah-LAH) and Classical (Tajweed-approximating hard syllables — Quran → koor-AHN); (3) TTS engineering rules (Q→K, Dh→TH, Kh→KH guttural, stress-cap, hyphen-glottal) promoted to numbered imperatives in content/_shared/arabic/01-tts-pronunciation-key.md; (4) per-book pronunciation.md gains a Mode column. The protocol is consulted on every authoring run; until B1–B8 land, the protocol is advisory and the rule files win. R-PHONETICS-OUT stays intact. Challenger Loop C5 will gain a sub-check flagging scripture-citation terms misclassified as Conversational.

·· the empirical feedback loop ··

audit_transcript.py — sensor end of the architecture

A new script closes the empirical-feedback loop. Reads a NotebookLM transcript and writes a per-episode markdown audit report. The report becomes the regression artifact against which the next upload is measured.
Sensor → analyser → report · the loop that was missing before 2026-05-17
·· baseline measurement · 2026-05-17 ··

Running audit_transcript.py against the 5 pre-v3.5 transcripts produced the empirical case for the architecture pivot: EP01 had 5 P0 hits + 17 P1 hits; EP02 7 P0 + 25 P1; EP03 22 P0 + 33 P1 (3 phonetic doublings, 19 mangled-name occurrences); EP04 7 P0 + 31 P1; EP05 18 P0 + 20 P1 (the famous "Du'a, du'a" / "Sahih Sitta, sahasita" closing-prayer episode). These baseline reports live in content/podcast/ayyuhal-walad/_system/audit-EP*.md as the diff target for the next NotebookLM upload after re-authoring under v3.5.

·· closed-loop learning · 2026-05-18 ··

The _learning/ substrate and the podcast-trainer agent

The challenger catches problems in one chapter. The substrate accumulates problems across every chapter. The trainer turns recurring problems into rule changes — but only after a regression suite says the change does not break any prior fix.

Until 2026-05-18 the quality system stopped at the challenger: every finding lived briefly in BOOK_DIR/_system/challenger-report.md, then was overwritten by the next run. There was no cross-book memory. A signature like TX-MANGLE:Tasawwuf->tassel wolf could keep appearing in book after book and never trigger a rule update unless a human noticed. The closed-loop substrate fixes that by giving every finding a permanent home and a deterministic path to promotion.

·· five stages · sense → aggregate → propose → test → promote ··

Sense. Every challenger pass and every audit_transcript.py run appends one JSONL record per finding to _learning/findings.jsonl via scripts/podcast/_rules.py::emit_finding(). Stable signatures, deduped per run, schema-versioned.

Aggregate. scripts/podcast/learn_aggregate.py groups the ledger by signature and writes _learning/patterns.md — a flat roll-up plus an in-book-density table (≥ 3 firings of one check against one (book, episode) is a strong signal even before cross-book recurrence).

Propose. scripts/podcast/learn_propose.py emits one markdown proposal under _learning/proposals/ for every signature that crosses a threshold (≥ 2 books OR ≥ 3 episodes OR ≥ 3 density). Routing tables pre-fill the candidate target file (handbook section / _rules.py constant / per-book mangle-map). Proposals that were already promoted (tombstone in _learning/promoted/) or already rejected (tombstone in _learning/archive/) are silently skipped. Records whose resolution is auto-fixed stop re-firing.

Test. scripts/podcast/test_challenger.py runs the deterministic auto-fix detectors against frozen fixtures at _learning/fixtures/<check-id>/{input.txt,expected.json}. Seven bootstrap fixtures shipped: B5 (em-dash), O1 (honorific repeat), N1 (inline phonetic), M3 (modernize), R4 (formal transition), J2-bold-header, R-NOSURPRISE-positive. Any rule change must keep all seven green.

Promote. The trainer agent applies the diff in a clean worktree, re-runs the regression suite, and on green commits the rule change on the book branch with a [trainer] tag and moves the proposal to _learning/promoted/<slug>.md with the commit hash. On red the proposal moves to _learning/archive/ with a Rejected because… line.

·· substrate layout ··

What lives under content/podcast/.skill/_learning/

·· append-only ledger ··

findings.jsonl

Every finding ever emitted, one JSON object per line. Fields: ts, source (podcast-challenger | audit_transcript), source_version, book, episode, chapter, check_id, severity (P0/P1/P2), signature, file, line, context_excerpt (≤300 chars), resolution (flagged | auto-fixed | carried). Schema is the contract; producers write through a single helper so the shape never drifts.

·· derived view ··

patterns.md

Regenerated on every aggregator run. Two tables: one for distinct signatures (count · books · episodes · severity · check · first seen · last seen · proposer-eligible) sorted by occurrence, and one for in-book density (signatures that fire repeatedly in one artifact). Never hand-edited.

·· workspace ··

proposals/ → promoted/ · archive/

Markdown proposals named YYYY-MM-DD-<signature-slug>.md. Each carries: trigger metrics, sample excerpts, candidate rule text, target file(s), required fixture additions, acceptance criteria, decision checkboxes, and a decision log. On accept → move to promoted/ with commit hash; on reject → move to archive/ with reason. The proposer skips signatures with a matching slug in either dir.

·· regression suite ··

fixtures/<check-id>/

One directory per check, each holding input.txt (a minimal artifact exhibiting the failure) and expected.json (the deterministic detector's expected hits). test_challenger.py reads both, runs the named detector, asserts equality. --update-golden regenerates expectations from current detector output for intentional rule shifts.

·· trend snapshot ··

health/<book-slug>.json + BOOK_DIR/_system/health-trend.md

Written by scripts/podcast/write_health.py at the end of every challenger run. Score formula 1 − (P0·1.0 + P1·0.2 + P2·0.05) / chapters_in_scope; badges Stable (≥ 0.95 for two runs) · Healthy (≥ 0.90) · Drifting · Unstable. The per-book trend file is append-only.

·· version stamp ··

scripts/podcast/_rules.py::CHALLENGER_VERSION

Single source of truth (currently 2.0) stamped into every challenger-report.md and every JSONL record's source_version. Anyone editing the challenger spec or _rules.py normative lists bumps this by 0.1 in the same commit so the ledger's version field stays meaningful for trend analysis.

·· the trainer agent ··

How podcast-trainer works

The challenger validates a chapter against the spec. The trainer changes the spec when the substrate says a pattern is recurring — but only when a regression suite proves the change does not break any prior fix. Five steps, one promotion ladder, INVARIANT walls.
·· step 1 · read ··

Read the substrate

The trainer reads _learning/patterns.md (not raw challenger-report.md files), every proposer-eligible proposal in _learning/proposals/, the per-book health/<book>.json, and the current state of every spec/handbook file in its writable set.

·· step 2 · hypothesize ··

Draft the smallest diff

For each candidate signature, draft the smallest possible edit to the targeted normative file (the proposal's Target file(s) line names it). Tag the change with the source_type from this book's contract so the diff is genre-scoped by default. List the citation excerpts that motivate it.

·· step 3 · test ··

Regression gate in a worktree

git worktree add a temp branch · git apply the diff · run python3 scripts/podcast/test_challenger.py. All seven fixtures must remain green. The live book branch is never polluted by experimental applies; the worktree is torn down whether the test passes or fails.

·· step 4 · promote ··

Commit on the book branch

On green: git apply the diff for real on book/<slug> · commit with subject podcast(challenger)[trainer]: <summary> · <n> occurrences in <book-slug>, passes regression · if the diff touches podcast-challenger.agent.md, also bump CHALLENGER_VERSION in _rules.py by 0.1 · move the proposal to _learning/promoted/ with the commit hash appended.

·· step 5 · flag ··

Archive failures · never commit

On red, on diffs touching an INVARIANT, or on P0-class proposals: move the proposal to _learning/archive/ with a Rejected because… reason. Append a row to the cumulative findings ledger noting the rejection. A regression FAIL is permanent for that diff — the trainer never tries the same diff twice; it may try a narrower diff that avoids the failing fixture.

·· anti-overfit walls ··

What the trainer must not do

· never edit _learning/fixtures/ (humans add anchors) · never touch the six INVARIANTS in skills-staging/podcast/SKILL.md §0 · never modify a *.py script · never collapse phase boundaries · never claim back the Phase 0f human gate · never commit on develop or main · never --no-verify a hook · never read other books' branches in a single pass.

·· promotion ladder ··

From observed pattern to landed rule

·· Tier 0 · Observation ··

1 occurrence

Stays in findings.jsonl only. No proposal, no commit. The signal is too thin to motivate a rule change.

·· Tier 1 · Candidate ··

≥ 3 occurrences (density · books · episodes)

learn_propose.py emits a proposal markdown. No commit yet — the trainer still has to apply, regression-test, and decide.

·· Tier 2 · Promoted (genre) ··

regression passes · single-genre

Trainer commits on the book branch with the [trainer] tag. The new rule lives inside a {source_type: <genre>} conditional block. The book's other chapters re-run challenger against the tightened rule before the book merges to develop.

·· Tier 3 · Cross-genre ··

≥ 2 books · ≥ 2 genres · regression everywhere

The trainer promotes the rule out of its genre block into the general spec. This typically takes multiple trainer passes across multiple books; the cumulative ledger in findings.jsonl is the evidence.

·· Tier H · Human-only ··

touches an INVARIANT · regression fails · P0-class

The trainer never auto-commits this class. The proposal is moved to _learning/archive/ with the reason; the human reviews and decides. P0 changes always require explicit human approval — never auto.

·· proposer noise suppression ··

v1.3 hardening

When every record for a signature carries resolution: auto-fixed, the proposer skips. Density-based proposals likewise only count still-flagged firings against the threshold. Once a rule lands, the matching pattern stops re-firing in subsequent proposer runs.

·· clarity wall · challenger vs trainer ··

The challenger validates one chapter. It reads the spec, applies the catalog, writes challenger-report.md, emits findings into the ledger, never edits the spec. The trainer changes the spec. It reads the ledger, drafts diffs, regression-tests, commits or archives — and never validates an individual chapter. The two agents touch different files and never overlap. The substrate is the shared persistence layer.

·· first promotion cycle · 2026-05-18 ··

The substrate ran one full sense → propose → promote cycle on its first day. Two proposals matured: (1) the J2 bold-header surface form proposal extended the R-NAMES auto-fix to cover markdown emphasis and section-header constructs (4 bio-anchors in kitab-al-riyad ch01 were rewritten to alias-led form); (2) the R-NOSURPRISE positive companion proposal added a "name what is new in ONE short clause" positive directive next to the existing DENY list (the DENY alone was empirically not enough — 5× firings of distinct surprise vocabulary in one transcript despite the DENY being present). Both promotions are now tombstones in _learning/promoted/ with their commit hashes. Health trend for kitab-al-riyad moved from 0.00 (Unstable) → 0.55 (Drifting) in the same cycle.

·· what enrichment is allowed from ··

The seven-tier source whitelist

Enrichment is what turns a translated section into a substantive standalone chapter. It comes from one of these seven tiers — never anywhere else.
Seven tiers, one cap, six anti-patterns · the enrichment is the difference between a raw source extract and a substantive episode
·· why all this rigor ··

NotebookLM reads literally. If the chapter has a sloppy citation, the hosts will repeat the sloppy citation. If the enrichment is shallow, the conversation will feel shallow. If the framing's tensions are generic, the conversation will drift. The quality system exists because we cannot edit the hosts after the fact — we can only steer them in advance.
