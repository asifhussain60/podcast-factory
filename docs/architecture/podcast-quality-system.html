podcast · quality system

Seven silent loops, a seven-tier pyramid, one challenger agent · empirical feedback closed

Quality control runs on four layers: seven silent loops every bundle passes through during authoring, a strictly tiered list of where enrichment is allowed to come from, an independent challenger agent with eleven categories of checks (A–O) that keeps iterating until the bundle is genuinely ready, and a new empirical feedback loop that audits the actual NotebookLM-generated transcript against every directive the customize prompt was supposed to enforce. The challenger reads two normative rule files as its authority: notebooklm-source-chapter-rules.md (17 chapter contracts) and notebooklm-customize-prompt-rules.md (12 framing contracts including welcome opening, anti-repetition, no-irrelevant-background, name aliasing, interruption avoidance, imperative-pronunciation, modernization DENY, surprise-noise DENY, and the no-read-aloud guard).

·· 2026-05-17 · architecture v3.5 ··

After a 5-transcript audit of Ayyuhal Walad exposed systematic failure modes (phonetic doublings like "Sahih Sitta, sahasita"; mangled names like "tassel wolf" for Tasawwuf; "the EI" for Ihya Ulum al-Din; 9 expanded honorifics in one episode; >40 surprise-noise injections across 5 episodes), the architecture pivoted: phonetic guidance moved from inline chapter parens (which NotebookLM reads aloud as content) to imperative customize-prompt directives (which the voice model uses silently). Three new chapter rules (R-PHONETICS-OUT, R-HONORIFIC-ONCE, R-NO-ABBREVIATION) and four new framing rules (R-PRONUNCIATION-IMPERATIVE, R-NOMODERNIZE, R-NOSURPRISE, R-NO-READ-PROMPT) make this enforceable. Three new challenger categories (M, N, O) and a new audit_transcript.py script close the empirical feedback loop.

·· four new loops · normative rules ··

The customize-prompt enforcement layer

Loops H, I, J, K each enforce a rule from notebooklm-customize-prompt-rules.md. The rule file is the single source of truth — both the producing skill and the challenger read it. No drift possible.
·· loop H · welcome + landing ··

Open with a welcome and summary; close on a question

The customize prompt must direct hosts to open with a brief welcome and a 2–3 sentence summary naming source + tension + landing question. Forbids "today we'll discuss" boilerplate. Closes on the unresolved tension or a single sharp line — never a recap of what was just said. Rules R-WELCOME + R-SUMMARYTAIL.

·· loop I · anti-repetition + background ··

Land each beat once; no Wikipedia detours

The customize prompt must forbid restating the central thesis more than twice, re-citing quotes, or summarizing what was just said. Biographical / historical background about author or translator appears only when pertinent and only once — never opening the episode, never returned to. Rules R-NOREPEAT + R-NOBACKGROUND.

·· loop J · name aliasing ··

Long name once, alias thereafter

"Imam Abu Hamid Muhammad al-Ghazali" said in full once, then "Ghazali" for every subsequent reference. "Hatim bin Ism al-Asamm" → "Haatim". ~30 canonical aliases live in content/_shared/arabic/05-name-alias-policy.md; both chapter and framing apply them. Rules R-NAMEALIAS + R-NAMES.

·· loop K · interruption avoidance ··

Each host completes a thought

The customize prompt must forbid mid-sentence interjections ("yeah", "right", "exactly") and talking-over. Hosts may pick up the thread after the other lands a beat, but not before. Concrete filler-word vocabulary is named in the framing so NotebookLM's voice model has a list to suppress. Rule R-NOINTERRUPT.

·· why externalize ··

Before the rule files existed, every rule had to be written twice — once in SKILL.md (so the producer obeys it during authoring) and once in the challenger (so the reviewer catches violations). Two copies = drift risk = silent rule-fork. Now both producer and reviewer read the same file. Add a new rule by adding a new section to the rule file; both pick it up next run.

·· three new categories · empirical-audit-driven ··

Categories M, N, O · added 2026-05-17

After the Ayyuhal Walad 5-transcript audit, three new check categories were added to the challenger. Each enforces a rule the soft-prose tone constraints could not.
·· category M · empirical-transcript loop ··

Modernization + surprise audit

The only challenger category that reads the actual NotebookLM transcript. Verifies the framing carries the canonical DENY blocks for modernization (Twitter, X, social media, algorithm, content creator, deep dive) and surprise noise ("wow", "it's chilling", "right?", "exactly"). Counts injections in the transcript and reports density per 1,000 words. Rules: R-NOMODERNIZE + R-NOSURPRISE.

·· category N · phonetic-as-content audit ··

Inline phonetics out → customize prompt in

Chapter contains zero inline (PHO-NE-TIC) parens (auto-fix stripped). Framing's ## Pronunciation block uses imperative form (auto-fix converts legacy passive lists). Every chapter Arabic term has a matching Pronounce "..." line in the framing. Framing ends with the no-read-aloud guard. Transcript scanned for doublings ("Sahih Sitta, sahasita") which signal residual inline phonetics. Rules: R-PHONETICS-OUT + R-PRONUNCIATION-IMPERATIVE + R-NO-READ-PROMPT.

·· category O · honorific + abbreviation audit ··

Honorific once · titles in full

Each honorific phrase form ("peace and blessings be upon him", "(PBUH)", "(AS)", "(RA)", "ﷺ") expanded at most once per chapter (auto-fix strips 2nd+ occurrences). No abbreviated work titles ("the Ihya", "EI", "the Nahj", "Sahihayn") — auto-fix replaces from the canonical map. Transcript scanned for the same patterns to confirm the chapter-level fixes propagated. Rules: R-HONORIFIC-ONCE + R-NO-ABBREVIATION.

·· the empirical feedback loop ··

audit_transcript.py — sensor end of the architecture

A new script closes the empirical-feedback loop. Reads a NotebookLM transcript and writes a per-episode markdown audit report. The report becomes the regression artifact against which the next upload is measured.
Sensor → analyser → report · the loop that was missing before 2026-05-17
·· baseline measurement · 2026-05-17 ··

Running audit_transcript.py against the 5 pre-v3.5 transcripts produced the empirical case for the architecture pivot: EP01 had 5 P0 hits + 17 P1 hits; EP02 7 P0 + 25 P1; EP03 22 P0 + 33 P1 (3 phonetic doublings, 19 mangled-name occurrences); EP04 7 P0 + 31 P1; EP05 18 P0 + 20 P1 (the famous "Du'a, du'a" / "Sahih Sitta, sahasita" closing-prayer episode). These baseline reports live in content/podcast/ayyuhal-walad/_system/audit-EP*.md as the diff target for the next NotebookLM upload after re-authoring under v3.5.

·· the second pair of eyes ··

The podcast-challenger agent

A separate agent that reads everything literally — exactly like NotebookLM does — and keeps iterating until the bundle is genuinely ready.
Three iterations max · auto-fix the deterministic · flag the semantic · ship when cleared
·· what enrichment is allowed from ··

The seven-tier source whitelist

Enrichment is what turns a translated section into a substantive standalone chapter. It comes from one of these seven tiers — never anywhere else.
Seven tiers, one cap, six anti-patterns · the enrichment is the difference between a raw source extract and a substantive episode
·· why all this rigor ··

NotebookLM reads literally. If the chapter has a sloppy citation, the hosts will repeat the sloppy citation. If the enrichment is shallow, the conversation will feel shallow. If the framing's tensions are generic, the conversation will drift. The quality system exists because we cannot edit the hosts after the fact — we can only steer them in advance.
