What each room can touch, and the two doors between them

Each skill writes only to its own area. There is one sanctioned door where the podcast may read a memoir chapter, and one shared brain that both rooms read from. Everything else is fenced off — by design, and by the actual code in the adapters.

·· why the walls ··

Three reasons the boundaries exist

·· reason 1 ··

The memoir has a voice

The journal skill's voice-fingerprint files are tuned over months of careful work. If the podcast skill could read them, it would imitate them — and the podcast voice should be NotebookLM's two hosts, not a memoir narrator. Different rooms keep different voices.

·· reason 2 ··

The podcast scratchpads change every day

If the journal skill could read podcast scratchpads, every memoir session would have to re-read potentially hundreds of unstable working files. The boundary keeps the memoir's context window calm.

·· reason 3 ··

One brain for what is genuinely shared

Pronunciation IS shared — the same Arabic term means the same thing in both rooms. So it lives in content/_shared/arabic/ and both rooms read from it. Everything else stays room-local.

·· enforcement ··

How the boundaries are actually enforced

SKILL.md boundary clauses

Each skill's SKILL.md has a Section that lists what it owns, what it reads, and what it does NOT touch. These are instructions the agent reads on every session start.

Adapter prohibited paths

The extract_chapter.py script (the only script that crosses the memoir/podcast boundary) carries explicit PROHIBITED_PATH_PREFIXES and PROHIBITED_NAME_PATTERNS lists. It will refuse to read a forbidden file even if the agent asks it to.

The challenger agent

The podcast-challenger agent is the second pair of eyes that catches boundary leaks in chapters and framings before they ship — meta-prose tells, cross-episode references, journal-file paths where they should not be.

Code review by Asif

The skills propose; Asif disposes. Every cross-skill writing attempt (proposed library entry, shared-manifest addition) lands in a staging file that Asif routes manually. No skill writes across a boundary without his approval.

·· the spirit ··

The boundaries are not bureaucracy. They are the reason this project can grow without each new skill destabilising the others. Add a third skill tomorrow — it gets its own room, reads the shared brain, and the existing rooms keep working untouched.
