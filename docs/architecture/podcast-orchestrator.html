Drop a book. Walk away. Come back to a shipped series — without two competing learning loops.

v1 of this plan (commit 596abe1) landed one commit after the closed-loop learning substrate shipped in 7ab3ac3 / 7e1afc1 / f41365e, and unknowingly invented a second one — separate ledger (trainer-ledger.md), separate regression suite (content/podcast/_system/regression-suite/), separate proposals dir, separate observations log, eight new state files per book. v2 collapses everything onto the existing content/podcast/.skill/_learning/ substrate: one ledger (findings.jsonl), one regression suite (_learning/fixtures/ + test_challenger.py), one proposals directory (_learning/proposals/ → promoted/), two state files per book (down from eight), one human gate (Phase 0f — chapter list + tier only; audience / angle / host dynamic are auto), three implementation passes (down from five). The orchestrator drives; the challenger validates; the trainer consumes the substrate. No new ledger formats. No parallel pipelines.

Note · what the Phase 0f gate actually reviews

The series-plan.md presented at the gate contains chapter list (titles + word-count targets) and length tier choice — the two outputs of Phase 0a–0e that benefit from human eyes (a bad segmentation wastes 8 chapters of work). The other three fields the chapter-contract template carries — audience, angle, host_dynamic — are not authoring decisions:
  • Audience · project-fixed in the orchestrator config (the listener profile is a property of the library, not the book)
  • Angle · project-fixed in the orchestrator config (the editorial stance is constant across the library)
  • Host dynamic · AI-selected per chapter from a canonical pair list in content/podcast/.skill/handbook/two-host-framing.md based on the chapter's content shape (curious_mind + patient_teacher · skeptic + advocate · etc.)
These three fields are written to series-plan.md for the audit trail and to the per-chapter contract for the build, but they do not invite human review.
·· convergence ··

Per-chapter loop · bounded by design

Three layers of cap. The inner cap is already enforced by the challenger. The middle cap is enforced by the orchestrator. The outer cap is the entire book — hard time + cost ceilings.
LayerCapEnforced byFailure mode
Inner (per challenger invocation)5 iterationschallenger_contract.max_iterations in podcast-challenger.agent.mdintelligent break · surface findings
Middle (per chapter)3 outer invocations_convergence.py in the orchestratorforce-ship SHIP-WITH-CAUTION · flag in challenger-report.md
Outer (per book)24h · $50orchestrator pre-flight + per-phase checkhalt cleanly · resumable

Per-outer-iteration decision rule

SHIP-READY                                   → break loop, ship
SHIP-WITH-CAUTION, iter ≥ 2                  → ship + flag (no P0)
SHIP-WITH-CAUTION, iter < 2                  → invoke fixer agent on P1s, retry
BLOCKED (any P0)                             → invoke fixer agent on P0s
                                               (max 3 fixer attempts per finding)
                                               re-author affected sections, retry
iter == 3 still BLOCKED                      → force-ship SHIP-CAUTION
                                               write into challenger-report.md
                                               continue to next chapter
                                               NEVER spin past 3 outer iterations

·· failure modes ··

What happens when something goes wrong

Every failure mode resolves to one of: STOP-with-fix-command, exp-backoff-then-skip, force-ship-with-flag, halt-cleanly-resumable.
FailureResponse
Pre-flight fail (Azure / dirty tree / wrong branch)STOP with exact fix command. No partial state.
Mid-book API outageexp backoff 3× (1s · 4s · 16s) per call → SKIP chapter on persistent fail → continue book
OCR garbledhalt before 0b · flag in orchestrator-state.json · leave branch in place
Killed mid-runstate file is the truth · --resume from last completed checkpoint
Git push failslog + retry every phase boundary · never --no-verify
Time cap (24h) or cost cap ($50) reachedhalt cleanly · partial-completion report · resumable
Convergence cap (iter == 3 still BLOCKED)force-ship SHIP-CAUTION · flag in challenger-report.md · next chapter
Trainer regression failarchive proposal with reason · never commit · ledger row appended
Trainer diff applies to INVARIANTarchive with Tier H — human only · never commit
·· out of scope ··

What this design explicitly does not do

  • Does not edit the regression suite contents (_learning/fixtures/) — humans add anchors
  • Does not touch the six INVARIANTS in skills-staging/podcast/SKILL.md §0
  • Does not skip the Phase 0f human gate · ever
  • Does not push to main · merges only to develop
  • Does not modify *.py from the trainer's autonomous side · trainer is a spec/handbook editor only
  • Does not run on memoir content — hard boundary
  • Does not read other books' branches in a single trainer pass · cross-book learning happens via cumulative findings.jsonl across trainer runs
