podcast · orchestration · v1 · 2026-05-18 · proposed

Drop a book. Walk away. Come back to a shipped series.

The existing /podcast skill is conversational — every phase waits on a human turn. The orchestrator overlay closes that loop for autonomous, multi-hour book processing: one command takes a PDF in _workspace/Books/ through Azure OCR, English refinement, phonetic indexing, chapter design, enrichment, framing authorship, episode compilation, and a bounded podcast-challenger convergence loop — emitting NotebookLM-ready bundles per chapter. There is exactly one human gate, at Phase 0f (Series Confirmation), where the chapter plan and series angle land for sign-off. Everything else runs hands-off, checkpoints to git every step, and reports progress in a file you can tail from your phone. When a book ships, a podcast-trainer agent reads the lessons across all of its challenger reports, proposes targeted improvements to the challenger spec and handbook, validates them against a held-out regression suite, and commits only what doesn't regress other genres. The book branch then merges to develop.

scope  phase 0a → final commit, hands-off  one human gate: phase 0f  no infinite loops · all caps explicit

·· diagram 2 of 4 ··

Convergence loop — bounded by design

The challenger already has a 5-iteration internal auto-fix loop. The orchestrator wraps a second outer loop that re-invokes the challenger after a fixer agent rewrites failing sections. Both loops have hard caps. A book can never spin forever.
·· why this cannot infinite-loop ··

Every loop in the orchestrator is bounded by a counter that lives in _system/orchestrator-state.json. The script reads the counter before each iteration; it exits the loop unconditionally when the counter hits its cap, regardless of verdict. The state file is written to disk before each loop body and is the single source of truth on resume — even a hard crash mid-iteration cannot reset a counter. There are six independent caps across the system (see the table below); the multiplicative ceiling per chapter is 25 challenger passes, per book is 25 × N chapters, and per run is the wall-clock guard and the Azure cost guard.

·· third surface · git log ··

Every phase boundary and every chapter ship emits an atomic commit on the book/<slug> branch. git log --oneline book/asaas-al-taveel from any machine shows the same picture: phase-by-phase progress, time-stamped, attributable. The trainer's own commits are tagged with [trainer] in the subject; chapter ships are tagged [ch01], [ch02], etc. The branch is pushed to origin after every commit so the state is visible on GitHub web while the run continues locally.

·· diagram 4 of 4 ··

The podcast-trainer agent — learning across books without regression

After all chapters in a book ship, the trainer reads the run's findings, looks for patterns recurring across chapters and books, proposes targeted edits to the challenger spec and handbooks, and gates every proposal through a held-out regression suite spanning multiple genres. Anything that improves only this book at the expense of others is rejected.
·· what the trainer can touch — and cannot ··

Editable targets, off-limits invariants

The trainer's reach is deliberately bounded. It can refine rules; it cannot rewrite invariants. It can add patterns; it cannot remove established gates.
·· in scope · trainer may propose edits ··

What the trainer can change

  • .github/agents/podcast-challenger.agent.md — refine rule descriptions, add new auto-fix patterns, tune severity thresholds
  • content/podcast/.skill/handbook/notebooklm-customize-prompt-rules.md — extend do-not patterns, must-do patterns
  • content/podcast/.skill/handbook/notebooklm-source-chapter-rules.md — refine chapter-level constraints when violations recur
  • content/_shared/arabic/03-arabic-english-manifest.md — propose new canonical entries (with attribution to the chapter where the term first appeared)
  • content/_shared/arabic/05-name-alias-policy.md — propose new alias rules (long → short forms)
·· out of scope · trainer cannot touch ··

What stays off-limits

  • The six INVARIANTS in SKILL.md §Section 0 — semantic guarantees of the architecture
  • The regression suite contents — only humans add or remove anchor chapters
  • Phase boundaries — the trainer cannot collapse, reorder, or skip Phase 0a–0g
  • The 0f human gate — autonomy can never claim back this gate
  • Script source (*.py) — only configuration / spec / markdown
  • Build invariants in build_episode_txt.py hard gates — these are code, not prompt
  • Anything outside content/podcast/ and content/_shared/arabic/
·· the promotion ladder ··

From observed pattern to landed rule

A finding doesn't become a rule on first sight. The trainer escalates evidence over time, and only the strongest, regression-clean proposals graduate to handbook text.
TierTriggerWhat the trainer doesWhere the change lands
Tier 0 Observation 1 occurrence in 1 chapter logs to _system/trainer-observations.md with citation; no action nowhere yet — pattern under watch
Tier 1 Candidate ≥ 3 occurrences across this book hypothesizes a refinement; writes proposal in _system/proposals/<pattern-id>.md book branch only · awaits regression
Tier 2 Promoted Regression passes AND single-genre auto-commits genre-tagged refinement on book branch handbook/* with {source_type: …} block
Tier 3 Cross-genre ≥ 2 books, ≥ 2 genres, regression passes everywhere promotes refinement out of genre block into general rule handbook/* general section
Tier H Human-only Touches an invariant, or regression fails, or P0-class change flags in _system/trainer-report.md · summary in completion notification nowhere — escalated to Asif for review

Phase A alone takes the human-coordination overhead away. Phase B closes Phase-0 manual gates. Phase C cuts framing authorship. Phase D closes the verdict loop. The trainer (Phase T) lands last because it needs a corpus of book branches to learn from.

·· deliberately deferred ··

Out of scope · explicitly

  • NotebookLM upload automation. No public API. Manual paste remains the human step at the end of every chapter — by design, not by oversight.
  • Multi-book parallelism. One book at a time. Book branches don't conflict, but the trainer must see a clean corpus per pass.
  • Loop M (post-publication transcript audit). Already specified in the skill (Section 4); orchestrator does not run it because TurboScribe is a manual subscription step. Trainer reads transcript audits when they exist.
  • Cross-skill training. The trainer reads only podcast-challenger and the podcast handbook. The memoir / journal pipelines are separately governed.
  • Reinforcement on rejected diffs. If the regression suite rejects 20 trainer proposals across 10 books, that's a signal the suite needs a human-curated update — not that the trainer needs more autonomy.
