Where the Arabic books turn into English

The /podcast skill needs faithful raw translations of Ismaili and Fatimid sources before it can build NotebookLM-ready episode bundles. Most of those sources are scanned Arabic PDFs with no embedded text. The Azure stack is the OCR and translation engine that converts them — four resources in one region under a single budget, fully scripted from a config file. 2026-05-18: provisioned end-to-end via infra/azure/provision-azure.sh; keys live in macOS Keychain; budget alerts wired to asifhussain60@gmail.com at 50% / 80% / 100% of a $50/month cap. Multi-Mac portability (via Azure Key Vault) is staged but not yet flipped on.

Four resources · one resource group · one region · one billing cap
·· the principle ··

The Azure stack does one thing well — convert scanned Arabic and Urdu PDFs into faithful English translations the /podcast skill can use. Every resource here exists to serve that workflow. Nothing more. The output is text, not the podcast itself; NotebookLM still does the audio.

·· what bit us ··

Pitfalls captured in troubleshooting.md

Every one of these was discovered during the 2026-05-18 setup. They're documented so you (or another Mac) don't re-discover them.
·· portal Marketplace ··

The third-party tile trap

Searching "Document Intelligence" in the Azure portal Marketplace surfaces third-party Azure Applications ("Document Intelligence Platform" by Stealth Labs LTD, "AI Easy Translator" by Sýkora IT, even Pig Latin translators) right next to the real Microsoft service. They look the same — they aren't. Fix: use deep links or az cli. Both are in docs/azure/troubleshooting.md.

·· shell syntax ··

zsh ≠ bash on read -s

macOS defaults to zsh. The bash idiom read -s -p "prompt" VAR errors with "read: -p: no coprocess". The zsh form is read -s "VAR?prompt". The provisioning scripts use #!/usr/bin/env bash so they're fine — only ad-hoc one-liners hit this.

·· storage names ··

Globally unique, no rename

Storage account names are globally unique across all of Azure, and they cannot be renamed after creation. We typoed once today (jounralpodcaststorage, the r and n swapped) — caught it before commit. The provisioning script does an availability check before az storage account create for this reason.

·· F0 vs S1 ··

Document translation needs Standard

Free F0 Translator supports text translation only. Document translation (the scanned-PDF Arabic→English workflow) requires Standard S1. If you accidentally provision F0, the API call returns 401 with a misleading "key invalid" message. TRANSLATOR_SKU="S1" in the config.

·· what the azure stack doesn't do ··

It does not write the podcast script. It does not pronounce Arabic. It does not host the journal app. It produces text only — page-anchored English translations of Arabic sources, plus a quality report. The podcast pipeline consumes that text. The Cloudflare tunnel handles the journal app's public access. The shared Arabic brain handles every pronunciation decision.
